# 实战配置对照手册

> 前情提要：前面 11 章分别讲了 RLinf 的架构、数据流、训练后端、算法、扩展点。本章换一个视角——不再逐个讲概念，而是把几种真实场景下的配置放在一起对比，看同一套框架在不同硬件、不同任务下，各个参数是怎么"跟着变"的。

## 一、为什么要横向对比配置

前面章节讲的每个参数，单独看都有清楚的含义（比如 `micro_batch_size` 受显存限制，`group_size` 决定要不要用组内比较）。但实际配一个新任务时，真正的难点是**这些参数要一起改，而且改的方向要互相匹配**——比如换成显存更小的 GPU，不只是 `micro_batch_size` 要调小，`enable_offload` 也大概率要打开，`is_lora` 可能也要跟着开。本章挑四种有代表性的真实配置，把它们放在一张表里，读者可以直接对照"我的场景更接近哪一种，该抄哪一组参数"。

## 二、四种场景速览

| | ManiSkill PPO + OpenVLA | LIBERO GRPO + Pi0 | IsaacLab PPO + GR00T | LIBERO Async PPO + Pi0.5 |
|---|---|---|---|---|
| 定位 | 入门验证 | 主流具身 RL 方案 | 大规模并行 | 高吞吐异步训练 |
| GPU 规模 | 2 卡，Actor/Env/Rollout 共享 | 8 卡，全部共享 | 8 卡，Actor 和 Env/Rollout 分开占用 | 8 卡，全部共享 |
| 核心算法 | PPO（`gae` + `actor_critic`） | GRPO（`grpo` + `actor`） | PPO | PPO + 异步 Runner |
| 模型规模 | OpenVLA 7B + LoRA | Pi0 3B 全参数 | GR00T | Pi0.5 + LoRA |

四种场景的选取标准不是随机的：前两种代表了"显存紧张、用 LoRA 省资源"和"显存够用、全参数训练"这两种最常见的资源约束；第三种代表 GPU 数量差异如何影响资源分配方式；第四种代表把同步模式换成异步模式需要额外打开哪些配置。逐个看。

## 三、场景一：ManiSkill PPO + OpenVLA——显存紧张时怎么办

### 3.1 约束条件决定了配置方向

这个场景只有 2 张 GPU，但要跑一个 7B 参数的 OpenVLA 模型，还要求 Actor（训练）、Env（仿真）、Rollout（推理）三种角色都要用到 GPU。约束很明确：**2 张卡塞不下"训练用的 7B 模型 + 推理用的 7B 模型 + 仿真渲染"这三者同时占用的显存**。这个约束直接推出了后面几个配置项：

```yaml
cluster:
  component_placement:
    actor,env,rollout: 0-1    # 三种角色共享同一组 2 张 GPU

actor:
  enable_offload: True        # 训练用完立刻把模型搬到 CPU，把 GPU 让给 Rollout/Env
  model:
    is_lora: True              # 只训练 LoRA 参数，优化器状态从 ~80GB 降到 ~1GB
    add_value_head: True       # PPO 需要 Critic

rollout:
  enable_offload: True         # 推理完同样让出显存
  pipeline_stage_num: 2        # Env 和 Rollout 交替用 GPU 时，用 pipeline 藏一部分等待时间
```

**这几项是绑在一起的一组决策，不是孤立的参数**：`component_placement` 把三种角色挤到同 2 张卡上，直接导致必须开 `enable_offload`（[第 06 章](./06_训练后端_FSDP与Megatron#六offload训练用完就把显存还回去)讲过的机制），否则同一时刻三份模型权重都想留在 GPU 上，显存根本不够。`is_lora: True` 则是从另一个方向缓解同样的显存压力——7B 模型全参数微调需要给每个参数额外存梯度和 Adam 的两个动量项，`is_lora` 把这部分开销从"整个模型的量级"降到"LoRA 秩对应的量级"，通常能差出一到两个数量级。三者共同的目标都是同一个：让 2 张卡也能跑起来。

### 3.2 数字上的直觉

`max_episode_steps: 80` 和 `total_num_envs: 32` 这两个值不是通用的默认值，是根据 ManiSkill 这个具体任务（一个较短的抓取任务）和 GPU 渲染的吞吐量定的——ManiSkill 用 GPU 做物理仿真，2 张卡还要分给训练和推理，能同时渲染的环境数天然受限，`32` 是这个约束下一个务实的选择，不是越大越好。

## 四、场景二：LIBERO GRPO + Pi0——显存够用时的"标准配置"

### 4.1 稀疏奖励决定了选 GRPO，而不是显存决定的

这个场景有 8 张卡，显存相对宽松（Pi0 只有 3B 参数），但选择用 GRPO 而不是 PPO 的原因和显存无关——LIBERO 这类任务的 reward 通常只有最终的 0/1（成功或失败），[第 09 章](./09_算法实现_GRPO配置详解#一grpo要解决什么问题)讲过，这种稀疏奖励下 Critic 很难学出准确的价值估计，GRPO 用"组内多次尝试互相比较"绕开了这个问题：

```yaml
algorithm:
  adv_type: grpo
  loss_type: actor
  group_size: 8
  rollout_epoch: 8
  filter_rewards: True
  logprob_type: chunk_level    # Pi0 的动作是 flow matching 产出，按 chunk 计 logprob 更自然

actor:
  micro_batch_size: 128         # 没有 value head，省下来的显存直接换成更大的 batch
  global_batch_size: 2048
  model:
    add_value_head: False
  fsdp_config:
    sharding_strategy: "no_shard"    # 3B 模型一张卡的显存完全够用，不需要真的切分
    gradient_checkpointing: False     # Pi0 的模型结构不支持梯度检查点

rollout:
  enable_offload: False        # 8 卡下 Actor 和 Rollout 不需要挤在一起，各自独立占用显存就够
  pipeline_stage_num: 1
```

### 4.2 no_shard 为什么不矛盾

[第 06 章](./06_训练后端_FSDP与Megatron#四sharding_strategyfsdp1独有的额外选项)提到过 `no_shard` 等价于退化成普通 DDP——这里选它并不是"没用 FSDP 的分片能力"，而是显存够用时，分片带来的 All-Gather/Reduce-Scatter 通信开销反而是纯粹的额外成本。8 张卡、每张卡都能独立装下一份完整的 3B 模型时，`no_shard` 是更快的选择，这正是前置知识里"如果模型本身不大，切分反而是多余开销"这句话在真实配置里的体现。

### 4.3 `filter_rewards` 为什么在这里几乎是必选项

`group_size: 8` 意味着每个任务跑 8 次——但 LIBERO 里有些任务对当前策略来说已经"太容易"（8 次几乎全成功）或者"太难"（8 次几乎全失败），[第 09 章](./09_算法实现_GRPO配置详解#三filter_rewards踢掉没有区分度的组)讲过这种组的优势估计没有意义，`filter_rewards: True` 把这些组从训练信号里过滤掉，把训练预算留给真正有区分度的那些任务。这不是可选的微调项，而是 GRPO 在这类任务上能不能训出来的关键开关。

## 五、场景三：IsaacLab PPO + GR00T——GPU 数量宽松时怎么分配

### 5.1 从"共享"变成"专用"

前两个场景 Actor/Env/Rollout 都挤在同一组 GPU 上，靠 offload 机制轮流用。这个场景有 8 张卡，而且 IsaacLab 的物理仿真本身就需要吃掉大量 GPU 资源（同时跑 4096 个并行环境），这时候更合理的选择是把 Actor 和 Env/Rollout **分开占用不同的物理 GPU**，谁都不用等谁把显存让出来：

```yaml
cluster:
  component_placement:
    actor: 0-3               # Actor 独占 4 张卡
    env,rollout: 4-7         # Env 和 Rollout 共享另外 4 张卡

env:
  train:
    total_num_envs: 4096     # IsaacLab 用 GPU 并行物理仿真，可以跑到这个规模
    max_episode_steps: 200

actor:
  micro_batch_size: 64
  global_batch_size: 4096    # 和环境规模匹配——环境越多，一步能收集的数据也越多
  model:
    num_action_chunks: 16    # GR00T 预训练时就是按 16 步一个 chunk 设计的
  enable_offload: False       # 独占 GPU，不用跟谁抢显存，不需要 offload
```

`enable_offload: False` 在这里不是"忘了开"，而是这个部署方式下 Offload 机制根本没有存在的必要——[第 06 章](./06_训练后端_FSDP与Megatron#六offload训练用完就把显存还回去)讲过 Offload 的意义是"轮流使用同一批 GPU 时把显存让出来"，Actor 和 Env/Rollout 既然已经分处不同的物理 GPU，就没有"轮流"这件事,Offload 反而是不必要的额外搬运开销。

### 5.2 num_action_chunks 为什么是 16 而不是 5

前两个场景用的 Pi0/OpenVLA，`num_action_chunks` 是 5 或 7；这里的 GR00T 用的是 16。这不是随意选的，而是 [第 11 章](./11_环境接入与模型适配#34-numactionchunks一次推理产出几步动作)讲过的——这个数字是模型架构本身设计决定的，接入新模型时要按模型预训练时用的 chunk 长度来配置，不是一个可以随便调的超参数。

## 六、场景四：LIBERO Async PPO + Pi0.5——从同步切到异步要多改什么

### 6.1 只多了一个开关，但背后的机制变了

前三个场景都用同步 Runner，这个场景换成异步模式（[第 07 章](./07_Runner训练循环#三异步模式不再互相等待)），目的是让 Actor 训练和 Env/Rollout 采集数据完全并行，不互相等待。从配置上看，改动出乎意料地小：

```yaml
actor:
  sync_weight_no_wait: True        # 权重同步改成"发出去不等待完成"

algorithm:
  loss_type: actor_critic          # 算法本身还是 PPO，没有变
  importance_sampling_fix: True    # 因为数据可能不是最新策略产出的，需要额外的修正
```

`sync_weight_no_wait: True` 直接对应 [第 07 章](./07_Runner训练循环#32-权重同步也要变成不等待否则白做了异步)讲过的 `no_wait` 机制——如果不开这个开关，虽然用了异步 Runner，权重同步那一步仍然会阻塞 Actor，相当于"名义上异步、实际上还是同步"，白白引入了异步模式的复杂度却没拿到并行的好处。

### 6.2 importance_sampling_fix 为什么在异步模式下几乎必须开

`importance_sampling_fix: True` 呼应了 [第 07 章第 3.3 节](./07_Runner训练循环#33-数据是旧策略产出的怎么修正)讲过的版本追踪机制——异步模式下 Rollout 用的策略权重可能落后 Actor 好几个版本,如果不做修正,直接把这些"过时"数据当成当前策略产出的数据去算 PPO 的 ratio,这个 ratio 在数学上就是错的（分母用的旧策略实际上比配置里假设的更旧）。开启这个开关后,训练流程里会额外做一次"重新计算 logprob"（recompute_logprobs），把 rollout 阶段记录的旧 logprob 和 Actor 重新算出来的 logprob 做对比,修正掉版本差异带来的偏差。同步模式下不需要这个开关，因为数据永远是最新策略产出的，没有版本差异需要修正。

## 七、从这四个场景里能抽出的通用决策逻辑

对照完四个场景,可以总结出一套"改配置"时的思考顺序,而不是死记某个具体数值：

1. **先看硬件约束**：GPU 数量、显存大小决定了 Actor/Env/Rollout 是共享 GPU 还是分开占用——共享就要开 `enable_offload`，分开就不需要。
2. **再看奖励的稀疏程度**：稠密奖励用 PPO（配 Critic），稀疏奖励（只有 success/fail）优先考虑 GRPO（配 `group_size` + `filter_rewards`）。
3. **模型规模决定 FSDP 分片策略和是否要 LoRA**：模型大、显存紧张 → 开 `is_lora`，考虑 `full_shard`；模型小、显存宽松 → `no_shard` 更快，可以全参数训练。
4. **`num_action_chunks` 等模型结构相关参数，照抄模型预训练时用的值**，不属于可以自由调整的超参数。
5. **同步 vs 异步的选择看 Actor 和 Rollout 是不是共享同一批 GPU**——共享的话异步模式没有实际并行收益，不如同步模式简单可靠；分开占用不同 GPU 时异步才能真正提升吞吐，但要记得配上 `sync_weight_no_wait` 和 `importance_sampling_fix`。

## 八、常见问题排查

| 症状 | 可能原因 | 对照本章哪个场景的做法 |
|------|---------|---------|
| OOM | 模型太大 / batch 太大 | 参考场景一：开 `enable_offload`、开 `is_lora`、减小 `micro_batch_size` |
| 训练很慢，GPU 利用率低 | Actor 和 Rollout 严格串行等待 | 参考场景四：如果硬件允许分开占用 GPU，考虑切换异步模式 |
| GRPO 训了很久 loss 不降 | 大量组全成功或全失败，没有有效信号 | 参考场景二：确认 `filter_rewards: True` 已开启，检查任务难度是否匹配当前策略水平 |
| 异步模式下训练不稳定、metrics 剧烈震荡 | 权重同步跟不上，off-policy 偏差过大 | 参考场景四：确认 `importance_sampling_fix: True`，或减小 `weight_sync_interval` 让同步更频繁 |
| 换了新模型后动作诡异 | `num_action_chunks` 配错，或旋转表示转换出错 | 参考 [第 11 章](./11_环境接入与模型适配#四动作格式转换模型输出和环境输入经常不是一套坐标系) |

## 系列总结

这十二章按"从外到内、从概念到实操"的顺序，覆盖了 RLinf 的整体架构（Worker 体系、调度系统）、数据如何流转、训练后端如何管理显存和分布式、几大类算法的设计取舍，最后落到怎么根据实际的硬件和任务去配置一次真实的训练。建议接下来找一个和自己场景最接近的配置（比如显存有限就参考场景一），实际跑起来，遇到问题时回来对照相应章节——概念性的理解和真正跑通一次训练之间，往往还需要几次踩坑才能补齐。
