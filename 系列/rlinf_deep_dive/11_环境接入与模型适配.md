# 环境接入与模型适配

> 前情提要：前面几章讲的是 RLinf 内部已经支持好的各种算法和训练后端。本章换个角度——如果你想接入一个全新的仿真器，或者接入一个 RLinf 还没支持的 VLA 模型，需要改哪些地方？

## 一、为什么需要"注册"这一层

RLinf 已经支持了十几种仿真环境（LIBERO、ManiSkill、RoboTwin、IsaacLab……）和十几种 VLA 模型（OpenVLA、π₀、GR00T……），但 Env Worker 和 Rollout Worker（[第 04 章](./04_Worker体系_五大角色详解)讲过）的代码不可能为每一种环境/模型各写一套——它们的代码是通用的，只在配置里写一个字符串（比如 `env_type: libero`），运行时"查表"找到对应的类去实例化。这个"查表"的机制就是本章要讲的注册系统：环境这边是 `get_env_cls()`，模型这边是 `get_model()`。

理解了这套机制，接入新环境/新模型就变成了一件很机械的事：写一个符合接口的类，在查表函数里加一行，剩下的调度、数据流转全部自动复用。

## 二、环境接入：一张"类型字符串 → 类"的映射表

### 2.1 get_env_cls：按需导入，而不是一次性全部导入

`rlinf/envs/__init__.py` 里的 `get_env_cls()` 就是这张映射表。它长这样：

```python
class SupportedEnvType(Enum):
    MANISKILL = "maniskill"
    LIBERO = "libero"
    ROBOTWIN = "robotwin"
    ISAACLAB = "isaaclab"
    # ... 还有 METAWORLD、CALVIN、ROBOCASA、REALWORLD 等十几种

def get_env_cls(env_type: str, env_cfg=None):
    env_type = SupportedEnvType(env_type)
    if env_type == SupportedEnvType.MANISKILL:
        from rlinf.envs.maniskill.maniskill_env import ManiskillEnv
        return ManiskillEnv
    elif env_type == SupportedEnvType.LIBERO:
        from rlinf.envs.libero.libero_env import LiberoEnv
        return LiberoEnv
    # ...
```

值得注意的一个细节：`import` 语句写在每个分支**内部**，而不是在文件顶部统一导入。这不是随意的写法——每种仿真器往往依赖一套独立的、体积不小的第三方库（比如 IsaacLab 需要装 Isaac Sim，ManiSkill 需要装 SAPIEN）。如果在文件顶部统一 `import` 所有环境模块，哪怕你这次训练只用 LIBERO，Python 也要在启动时把 IsaacLab、ManiSkill 等所有依赖库都加载一遍——不仅慢，而且只要有一个环境的依赖没装，整个程序都跑不起来。写在分支内部实现了**按需加载**：只有你实际配置要用哪个环境，才会真正 `import` 它对应的那些依赖。

### 2.2 环境类要实现的接口

`get_env_cls()` 返回的类需要满足一套类似 Gymnasium 的接口——`reset()` 给初始观测，`step()` 执行动作、返回下一步的观测和奖励：

```python
class BaseEnv:
    def reset(self) -> dict:
        """返回初始观测：{"main_images": tensor, "states": tensor, "task_descriptions": list}"""

    def step(self, actions: torch.Tensor) -> tuple:
        """actions: [num_envs, action_dim]
        返回: (obs, rewards, dones, infos)"""
```

这里的观测字典格式要和 [第 05 章](./05_数据流与通信机制#env-output环境产出)讲过的 `EnvOutput` 对齐——`main_images`、`states`、`task_descriptions` 这些字段名是 Rollout Worker 和策略模型约定好的输入格式，新接入的环境类必须按这个格式组织返回值，下游的模型推理代码才不需要为每种环境单独写适配逻辑。

### 2.3 接入一个新环境的四步

1. 在 `SupportedEnvType` 里加一个新的枚举值
2. 实现环境类，继承基类接口，把仿真器原生的 API（不管是 SAPIEN、MuJoCo 还是别的）包装成 `reset()`/`step()` 的形式
3. 在 `examples/embodiment/config/env/` 下写一份该环境的 YAML 配置（任务名称、图像分辨率、动作维度等仿真器特有的参数）
4. 在 `get_env_cls()` 里加一个分支，把新的枚举值映射到刚写的类

四步做完，Env Worker、Runner、数据流转这些通用逻辑完全不需要改——它们只认 `reset()`/`step()` 这套接口，不关心背后是哪个仿真器。

## 三、模型接入：同样的注册思路，但接口更复杂

### 3.1 get_model：按模型类型分发，还要处理 LoRA 注入

模型这边的入口是 `rlinf/models/__init__.py` 里的 `get_model()`，思路和环境注册一致（按类型字符串分发、按需导入），但多做了一步——处理 LoRA 的注入（如果配置里 `is_lora: True`）：

```python
def get_model(cfg):
    model_type = get_supported_model(cfg.model_type)
    if model_type == SupportedModel.OPENVLA:
        from rlinf.models.embodiment.openvla import get_model
    elif model_type == SupportedModel.OPENPI:
        from rlinf.models.embodiment.openpi import get_model
    elif model_type == SupportedModel.GR00T:
        from rlinf.models.embodiment.gr00t import get_model
    # ...
    model = get_model(cfg, torch_dtype_from_precision(cfg.precision))

    if cfg.is_lora:
        # 用 peft 库的 LoraConfig 给模型注入 LoRA 层
        model = get_peft_model(model, lora_config)
    return model
```

**为什么 LoRA 注入放在这个统一的入口，而不是让每个模型自己处理**：不管底层是 OpenVLA 还是 π₀，"给某些线性层加 LoRA 旁路"这件事的逻辑是通用的（[LoRA 前置知识](/前置知识/000x_前置知识_LoRA低秩适配基础)讲过原理），只有 `target_modules`（哪些层需要加）会因模型结构不同而略有差异。把这段逻辑收在 `get_model()` 里统一处理，每个具体模型的实现文件就不需要重复写一遍 LoRA 注入代码。

### 3.2 BasePolicy：策略模型必须实现的三个方法

每个具体模型（`OpenVLAPolicy`、`OpenPiPolicy` 等）都要继承 `BasePolicy`，这个基类用抽象方法的方式**强制**规定了两个必须实现的接口，外加一个统一的调度入口：

```python
class BasePolicy(ABC):
    def forward(self, forward_type=ForwardType.DEFAULT, **kwargs):
        if forward_type == ForwardType.DEFAULT:
            return self.default_forward(**kwargs)
        # 其他 forward_type（SFT、SAC、NFT...）分发给对应的方法

    @abstractmethod
    def default_forward(self, **kwargs): ...    # Actor 训练时调用：给定动作，算 logprob/entropy/value

    @abstractmethod
    def predict_action_batch(self, **kwargs): ... # Rollout 推理时调用：给定观测，采样出动作
```

**为什么 `forward()` 要按 `forward_type` 分发，而不是每种训练方式单独开一个入口方法**：回顾 [第 10 章](./10_算法实现_SAC与其他算法)，SAC、DAgger、NFT 这些算法需要模型支持不同的计算模式（SAC 需要算 Q 值，DAgger 需要监督学习式的前向，NFT 需要对去噪的每一步单独算能量）。如果每种算法都要求模型类实现一个新的独立方法名，Actor Worker 调用模型时就要为每种算法写不同的调用代码，非常繁琐。统一成 `forward(forward_type=...)` 这一个入口后，Actor Worker 不用关心具体是哪种算法，只需要把对应的 `forward_type` 传进去，模型内部自己分发到 `sft_forward`、`sac_forward` 等方法——这些方法默认抛出 `NotImplementedError`，只有真正支持这种训练方式的模型才需要覆写。

`predict_action_batch()` 则是纯粹给 Rollout Worker 用的推理接口——输入观测，输出这一步（或这一个 action chunk）该执行的动作，同时（如果算法需要）返回这次采样对应的 logprob、value 等信息，格式要能被组装进 [第 05 章](./05_数据流与通信机制#rolloutresult推理产出)讲过的 `RolloutResult`。

### 3.3 接入一个新模型的四步

1. 在 `SupportedModel` 枚举里加一个新值
2. 实现一个继承 `BasePolicy` 的类，至少要写好 `default_forward()`（训练用）和 `predict_action_batch()`（推理用）；如果这个模型要支持 SAC/DAgger/NFT 等特殊算法，再额外覆写对应的 `xxx_forward()`
3. 在 `examples/embodiment/config/model/` 下写模型配置——至少要包含 `model_path`、`precision`、`num_action_chunks`、`action_dim` 这几个通用字段（[第 08 章](./08_算法实现_PPO配置详解)、[第 09 章](./09_算法实现_GRPO配置详解)里出现的很多算法参数都是通过这几个字段和模型对接的）
4. 在 `get_model()` 里加一个分发分支

### 3.4 num_action_chunks：一次推理产出几步动作

模型配置里有一个所有 VLA 模型通用的关键参数：

```yaml
model_type: openpi
num_action_chunks: 5   # 一次 predict_action_batch() 产出 5 步动作
```

VLA 模型通常不是"看一次图算一个动作"，而是"看一次图，一次性预测接下来连续 5 步的动作序列"（这样能减少推理频率，也更符合很多模型架构本身的设计，比如扩散/flow matching 类模型天然产出一个动作序列）。这个参数直接决定了 [第 08 章](./08_算法实现_PPO配置详解#四rolloutepoch-与-groupsize-如何共同决定数据量)提到的 `n_chunk_steps` 怎么算——比如 `max_steps_per_rollout_epoch=240`、`num_action_chunks=5`，一个 rollout epoch 就产出 `240/5=48` 个 chunk step，这个数字会一路传递到后面 GAE 计算、batch 切分等环节。

## 四、动作格式转换：模型输出和环境输入经常不是一套坐标系

### 4.1 为什么需要专门的转换层

模型输出的动作和仿真器/真机期望的动作格式往往不一致：模型可能输出归一化到 `[-1, 1]` 的值，环境需要真实的物理量（比如米、弧度）；模型可能用 6D rotation 表示旋转（对神经网络训练友好，没有欧拉角的万向锁问题），但环境接口只认四元数或欧拉角。`rlinf/envs/action_utils.py` 里的 `prepare_actions()` 就是专门处理这层转换的——把模型输出"翻译"成环境能听懂的格式，包括反归一化、旋转表示转换、动作范围裁剪。

### 4.2 旋转表示为什么有这么多种

具体到旋转这一项，`rlinf/envs/rotation_utils.py` 提供了几种表示之间的相互转换：

| 表示 | 维度 | 为什么需要它 |
|------|------|------|
| Euler（欧拉角，rpy） | 3 | 人类最容易理解和调试，但存在万向锁问题（某些姿态下会丢失一个自由度） |
| Quaternion（四元数） | 4 | 大多数仿真器/机器人接口的标准输入格式，没有万向锁 |
| 6D rotation | 6 | 对神经网络训练更友好——是旋转矩阵前两列展开，输出空间连续，不像四元数那样存在双重覆盖（$q$ 和 $-q$ 表示同一个旋转） |
| Axis-angle（轴角） | 3 | 部分仿真器的原生格式 |

新接入一个环境时，如果它的动作接口用的旋转表示和模型输出的不一致（大概率不一致），就需要在 `prepare_actions()` 里补上对应的转换逻辑——这也是接入新环境时最容易被忽略、但出错了很难排查的一个环节（旋转转错了不会报错，只会让机器人的动作诡异地偏转）。

## 五、真机环境：接口约定不变，只是 step() 里换成了真实硬件调用

`rlinf/envs/realworld/` 提供了真机 RL 的环境实现，它仍然遵守 `reset()`/`step()` 的统一接口，区别只在于内部实现——`step()` 不再是调用仿真器的物理引擎，而是通过 ROS2 或厂商 SDK 把动作指令发送给真实的机械臂，等待执行完成后从真实相机（RealSense/ZED 等）和关节编码器读回观测：

```python
class RealWorldEnv:
    def step(self, actions):
        # 通过 ROS2/自定义通信协议，把动作发送给真实机械臂
        # 等待机械臂执行完成
        # 从真实相机 + 关节状态读取新的观测
        # 计算奖励（真机场景下奖励通常来自规则判断或人工反馈，很少有稠密的仿真奖励）
```

正是因为 `reset()`/`step()` 这套接口在仿真和真机之间是统一的，前面几章讲的 Runner、Actor、Rollout 的调度逻辑不需要为"真机训练"这个场景做任何特殊处理——唯一的差异隔离在环境类内部,以及真机场景下奖励通常更稀疏这个事实,后者会影响你在配置层面选择哪种算法（[第 10 章](./10_算法实现_SAC与其他算法)里 SAC/DAgger 这类不完全依赖稠密奖励的算法在真机场景更常见）。

## 六、总结

| 扩展点 | 入口函数 | 必须实现的接口 | 关键坑 |
|--------|---------|--------------|--------|
| 新环境 | `get_env_cls()` | `reset()` / `step()`，观测字段名要对齐 `EnvOutput` | 按需 import，避免拖入不需要的依赖 |
| 新模型 | `get_model()` | `default_forward()` / `predict_action_batch()` | 特殊算法（SAC/DAgger/NFT）需要额外覆写对应 forward |
| 动作格式 | `prepare_actions()` | —— | 旋转表示转换容易被忽略,出错也不易发现 |
| 真机 | 继承同样的 `reset()`/`step()` 接口 | —— | 奖励通常稀疏，算法选择要相应调整 |
