# 双臂任务训练方法研究笔记

本文关注一个具体问题：做双臂协作任务时，模型训练是不是只要把单臂的 state 和 action 扩成左右臂两份？

结论先说清楚：这只能作为最低 baseline。双臂任务真正多出来的不是几个 action 维度，而是两只手之间的耦合关系。很多失败不是因为单只手不会动，而是因为两只手的节奏、角色、接触约束和物体状态没有对上。

如果把双臂当成“两个单臂拼起来”，模型要从数据里自己摸索这些东西：

- 左手和右手什么时候同步，什么时候一先一后。
- 哪只手负责稳定物体，哪只手负责操作。
- 两个末端和物体之间的相对位姿约束。
- 接触发生后，动作目标应该从“移动手”变成“控制物体运动”。
- 一只手的动作会改变另一只手看到的状态和可行动作空间。

这些关系如果不显式建模，通常会导致数据利用率很差。下面按研究方向讲清楚代表性工作、核心原理、为什么能省数据，以及对 embodied-arena 这类仿真平台应该怎么落地。

## 1. 最低 baseline：直接拼接 state/action

最朴素的做法是：

$$
obs = [image, left_arm_state, right_arm_state, object_state]
action = [left_arm_action, right_arm_action]
policy(obs) -> action
$$

这类方法的好处是实现简单，也公平：它回答了“只增加 action/state 维度能不能解决问题”。如果你要做论文或系统评测，这个 baseline 必须有。

但它的问题也很明显。模型看到的是一大串数，里面没有告诉它哪些量是“左手相对物体”、哪些是“右手相对左手”。如果数据量很大，模型可能能学出来；但如果示教数据少，它会把很多样本浪费在重新发现这些结构上。

例如开瓶盖任务里，左手的关键不是去某个绝对坐标，而是相对瓶身保持稳定夹持；右手的关键不是单独旋转，而是在左手约束下对瓶盖施加相对旋转。直接拼接 action/state 会把这个任务写成两条末端轨迹的拟合问题，丢掉了“瓶身-瓶盖-双手”之间的约束结构。

所以直接拼接可以作为 baseline，但不应该作为最终方法。

## 2. PerAct2：双臂需要显式 coordination

论文：PerAct2: Benchmarking and Learning for Robotic Bimanual Manipulation Tasks  
链接：https://arxiv.org/abs/2407.00278

PerAct2 的价值不只是提出一个模型，更重要的是它把双臂学习里的几种常见思路摆出来比较：

- 两个独立单臂 agent。
- leader-follower 结构。
- 单个双臂模型联合预测。
- 通过注意力机制建模两只手之间的交互。

两个独立单臂 agent 听起来很自然：左臂一个 policy，右臂一个 policy，各自看图、各自动作。但问题是 coordination 只能通过环境状态间接发生。左手不知道右手接下来想干什么，右手也不知道左手是否已经稳定住物体。对于双手搬物、插入、折叠、开盖这类任务，这种隐式通信太弱。

leader-follower 会好一些。一个主臂先做决策，另一个从臂跟随或补偿。它适合角色固定的任务，比如永远左手扶、右手操作。但真实双臂任务里，角色经常会切换：有时左手扶，有时右手扶；有时两只手共同搬运；有时一只手临时让位。固定 leader-follower 容易把任务假设写死。

PerAct2 更有启发的是：双臂模型应该共享感知，又要保留左右臂各自的动作头，中间通过 attention 或类似机制交换信息。可以把它理解成：

```text
image/object features -> shared scene representation
left arm token -> ask scene and right arm: 我该怎么动
right arm token -> ask scene and left arm: 我该怎么动
```

这比简单 concat 更强，因为模型内部有机会显式表示“左手动作依赖右手状态”。它也比两个独立 agent 更省数据，因为左右臂共享视觉理解，不需要各自学一套完整的感知系统。

对 embodied-arena 的落地建议：

- 保留 concat BC 作为最低 baseline。
- 做一个 shared visual encoder + left/right arm token 的模型。
- 在 token 之间加 cross-attention 或 transformer block。
- 输出仍然可以是 `[left_action, right_action]`，但中间表示不要只是拼接 MLP。
- 做 ablation：two independent policies vs joint policy vs cross-attention joint policy。

这个实验能回答一个核心问题：双臂收益来自 action 维度变大，还是来自模型学到了左右臂交互。

## 3. Stabilize to Act / BUDS：把双臂拆成“稳定手”和“操作手”

论文：Stabilize to Act: Learning to Coordinate for Bimanual Manipulation  
链接：https://arxiv.org/abs/2309.01087  
项目页：https://sites.google.com/view/stabilizetoact

这篇工作的出发点很人话：很多双手任务不是两只手都在做同一种事情，而是一只手先把世界变得稳定，另一只手再完成主要操作。

比如：

- 开瓶盖：一只手固定瓶身，另一只手旋盖。
- 切菜：一只手按住食材，另一只手拿刀。
- 拉拉链：一只手拉住衣物，另一只手拉拉链头。
- 插拔：一只手固定插座或物体，另一只手插入。
- 擦桌子上的物体：一只手压住或扶住，另一只手擦。

如果用一个端到端 policy 同时预测两只手动作，模型要自己学出“谁稳定、谁操作”。BUDS 直接把这个结构放进方法里：先决定 stabilizing arm 应该在哪里接触、如何稳定物体，再让 acting arm 完成主要任务。

它的核心不是更大的网络，而是降低学习问题难度。原本模型要学：

```text
scene -> left hand full trajectory + right hand full trajectory
```

现在变成：

```text
scene -> stabilizing contact / stabilizing pose
scene + stabilized object -> acting trajectory
```

这会更省数据，因为 stabilizing 行为通常比完整双臂轨迹简单。很多任务里稳定手只需要找到一个合理接触点，然后维持约束；操作手才需要细致轨迹。把两者分开后，每个子问题的数据分布更窄，示教数据更容易覆盖。

这篇工作报告了少量示教下的真实机器人双臂任务结果，说明角色分解确实能提高样本效率。更重要的是，它给了一个很适合平台化 benchmark 的建模角度：任务不是统一叫 bimanual，而是要区分 coordinated、symmetric、support-act、handover 等结构。

对 embodied-arena 的落地建议：

- 给任务配置增加可选的 role annotation：

```text
left_role: stabilizer | actor | both
right_role: stabilizer | actor | both
```

- 对 support-act 类任务，加 stabilizing target 监督：

```text
stabilizing_ee_pose
stabilizing_contact_point
stabilizing_contact_normal
acting_ee_trajectory
```

- 训练时可以用两个 head：

```text
stabilizer_head(scene) -> stabilizing pose/contact
actor_head(scene, stabilizer_context) -> acting action
```

- 评测时对比：

```text
joint BC
joint BC + role token
stabilizer-actor decomposition
```

这个方向尤其适合开盖、开抽屉、拉门、插拔、切割、清洁、衣物整理这类任务。

局限也要说清楚：不是所有双臂任务都能拆成稳定手和操作手。双手共同搬运托盘、双手拉伸布料、双手同步折叠，这些更像 symmetric coordination，而不是 support-act。

## 4. VoxAct-B：用语言和体素把“稳定哪里、操作哪里”讲清楚

论文：VoxAct-B: Language-Conditioned Bimanual Manipulation with Voxels  
链接：https://arxiv.org/abs/2407.04152  
代码：https://github.com/VoxAct-B/voxactb

VoxAct-B 可以看成 BUDS 思路的一个视觉语言版本。它不满足于让模型从图像里自己猜“哪里该扶、哪里该动”，而是借助视觉语言模型找任务相关区域，再用体素表示做双臂 action 预测。

这里的关键不是“用了 VLM 所以高级”，而是它把双臂任务拆成了更可解释的空间问题：

```text
语言指令 -> 找到任务相关区域
3D/voxel 表示 -> 表示物体和可操作空间
acting branch -> 预测操作手动作
stabilizing branch -> 预测稳定手动作
```

为什么体素有用？因为双臂操作本质上很空间化。手要去哪里、物体哪个部位要被固定、哪个部位要被拉开，这些关系在 3D 空间里比在扁平向量里更自然。体素表示让模型直接在空间网格中选择或评分，而不是从一串 state 里隐式恢复几何关系。

它对数据利用率的帮助主要来自两点：

- 语言/VLM 帮模型缩小关注区域，不用每次从整张图里学什么重要。
- acting/stabilizing 分支减少了完整双臂 action 的搜索空间。

这对 embodied-arena 的启发是：如果任务有语言目标或结构化任务描述，不要只把 language embedding 拼到 state 后面。更好的做法是让语言参与空间 grounding，比如指出目标物、可操作部位、稳定部位。

可落地的简化版本：

```text
task instruction -> target object id / affordance id
asset affordance metadata -> candidate acting region and stabilizing region
policy observes these region features
```

你们平台已经有 assetLab、affordance、task config 体系，比真实机器人更容易拿到 object id、link id、joint id、affordance region。可以先不用大 VLM，而是用仿真标注直接提供这些区域，再评估结构信息对数据效率的提升。

建议 ablation：

```text
no affordance
target object id only
acting affordance only
acting + stabilizing affordance
```

这个实验能说明：双臂任务里，显式告诉模型“哪里该扶、哪里该动”到底能省多少数据。

## 5. RoTri-Diff：不要只看手，也不要只看物体，要看三者关系

论文：RoTri-Diff: Robot-Object Triadic Diffusion for Bimanual Manipulation  
链接：https://arxiv.org/abs/2603.07165

这类工作的核心观点非常适合回答“state 到底该怎么加”：双臂任务里，最有用的 state 往往不是更多绝对坐标，而是相对关系。

普通 robot-centric 表示会关注：

```text
left_ee_pose in world
right_ee_pose in world
joint positions
```

普通 object-centric 表示会关注：

```text
object_pose
left_ee_pose relative to object
right_ee_pose relative to object
```

但双臂协作还缺一个关键关系：

```text
left_ee_pose relative to right_ee_pose
```

也就是说，真正重要的是一个三角关系：

```text
left hand <-> object
right hand <-> object
left hand <-> right hand
```

这就是 triadic interaction 的直觉。比如双手搬托盘时，两只手相对物体的位置决定了托盘是否平衡；两只手之间的距离和高度差决定了物体是否倾斜；如果只看每只手相对物体，模型可能仍然不知道两只手之间的协调姿态是否合理。

这种方法常和 diffusion policy 结合，因为双臂动作具有多模态性。相同目标可能有多种合法双手姿态：左手靠前右手靠后，或者两手对称握持；从左侧绕还是从右侧绕。普通 L2 行为克隆容易把多种动作平均成一个无效动作。diffusion 可以表示多个可能轨迹，而 triadic relation 给 diffusion 一个更好的结构化条件。

对 embodied-arena 的落地建议：

- 不要只记录 raw joint state，要在 observation 里加 relation features：

```text
T_left_object
T_right_object
T_left_right
distance_left_object
distance_right_object
distance_left_right
relative yaw / approach direction
contact flags
```

- 如果用 6D rotation 表示，可以避免 quaternion 符号翻转问题。
- 如果使用四元数，项目里必须保持 Isaac Lab 新顺序 `[x, y, z, w]`。
- 做 ablation：

```text
absolute ee poses only
absolute + hand-object relative poses
absolute + hand-object + hand-hand relative poses
```

这个方向的好处是很容易在仿真平台里做，因为所有 pose 都能直接从 scene 里拿到。它也很适合作为“单臂 vs 双臂”研究的核心变量：单臂没有 hand-hand relation，双臂有。

## 6. State Diffusion + Inverse Dynamics：先学物体怎么动，再学手怎么动

论文：Learning Coordinated Bimanual Manipulation Policies using State Diffusion and Inverse Dynamics Models  
链接：https://arxiv.org/abs/2503.23271

这篇工作的思路是把“预测动作”改成“两步走”：

```text
history state -> future state / object motion
future state + current state -> action
```

为什么这对双臂特别有意义？因为很多双臂任务的目标不是让手到达某个位置，而是让物体按照正确方式运动。

比如：

- 双手搬物：关键是物体平稳移动，而不是两只手各自轨迹长什么样。
- 拉开抽屉：关键是抽屉 joint position 变化，手的轨迹可以有多种。
- 折叠布料：关键是布料关键点的未来位置。
- 双手推 L 形物体：关键是物体位姿变化，不是手端点绝对轨迹。

如果直接监督 action，模型会被示教者的动作风格绑定住。同一个物体运动目标可能对应很多手部动作，action supervision 会把这些多样性当成噪声。先预测 future state，相当于先问：“我想让世界变成什么样？”再用 inverse dynamics 回答：“为了做到这个，我的两只手该怎么动？”

这通常能提高数据利用率，因为 future object state 比 raw action 更接近任务目标，也更容易跨示教者、跨初始位姿泛化。

在 embodied-arena 里可以做得更明确。仿真中可以拿到 object pose、articulation joint、关键点、接触状态，所以可以设计辅助预测目标：

```text
future_object_pose
future_articulation_joint_pos
future_keypoints
future_contact_state
```

训练结构可以是：

```text
encoder(obs_history) -> latent
state_head(latent) -> future object/task state
inverse_dynamics_head(latent, future_state) -> dual-arm action
```

也可以不完全照论文做 diffusion，先做一个简单版本：

```text
main loss: action BC
aux loss: predict future object pose / joint state
```

如果这个简单版本已经明显提升，就说明任务确实更依赖 object-centric future modeling。

适合任务：

- 双手搬运刚体。
- 开抽屉、开门、开盒子。
- 推拉有约束物体。
- 布料、绳子、柔性物体。
- 需要保持物体稳定的运输任务。

不适合或收益较小的任务：

- 目标几乎只由末端位置决定的 reach 类任务。
- 物体状态变化很小的纯抓取准备动作。

## 7. PPI：用 gripper keypose 和 object pointflow 做中间监督

论文：PPI: Gripper Keypose and Object Pointflow as Interfaces for Bimanual Robotic Manipulation  
链接：https://arxiv.org/abs/2504.17784  
项目页：https://yuyinyang3y.github.io/PPI/

PPI 的想法很实用：直接从图像到连续双臂动作太难，中间应该加一些人能理解、模型也更容易学的接口。

它用了两个中间表示：

```text
gripper keypose: 关键时刻左右夹爪应该在哪里
object pointflow: 物体表面点接下来怎么运动
```

keypose 解决“手该去哪里”的问题。pointflow 解决“物体应该怎么变”的问题。二者合起来，比单纯 action trajectory 更结构化。

为什么这能省数据？因为 action 序列很长，里面大量时间步只是控制细节。示教之间细节差异很大，但关键位姿和物体运动目标往往比较稳定。模型如果先学这些稳定结构，再生成连续动作，就不需要用有限数据去拟合所有低层细节。

举个例子：双手把盒盖打开。不同示教者的手腕轨迹可能不同，但关键点差不多：

```text
左手 keypose: 固定盒身某侧
右手 keypose: 接近盒盖把手
object pointflow: 盒盖绕铰链向上转
```

只监督 dense action 时，模型不知道这些中间目标；加 keypose 和 pointflow 后，学习目标变得更像任务本身。

对 embodied-arena 的落地建议：

- 在数据生成或 replay 时自动提取 keypose：

```text
first contact pose
stable grasp pose
pre-insertion pose
handover pose
release pose
```

- 对刚体，pointflow 可以用物体 mesh 上采样点的前后位姿变换得到。
- 对 articulation，pointflow 可以由 link pose 或 joint state 推导。
- 对 deformable，先用关键点 flow，而不是完整点云 flow。

训练时可以加 auxiliary heads：

```text
policy_encoder(obs) -> latent
action_head(latent) -> action chunk
keypose_head(latent) -> left/right future keypose
pointflow_head(latent) -> object point displacement
```

评测时不仅看 success，还看：

```text
keypose error
object future pose error
contact timing error
```

这样能更清楚地解释模型为什么成功或失败。

## 8. D-CODA：双臂数据增强必须保持“协同一致”

论文：D-CODA: Diffusion for Coordinated Dual-Arm Data Augmentation  
链接：https://arxiv.org/abs/2505.04860  
项目页：https://dcodaaug.github.io/D-CODA/

很多机器人学习方法会做图像增强，比如 crop、color jitter、加噪声。但双臂任务的问题不是图像不够多，而是 coordinated examples 不够多。也就是说，模型需要看到更多“左手和右手如何一起做对”的例子。

D-CODA 的关键是：增强双臂数据时，不能只改图像，还必须生成与图像一致、与接触约束一致的双臂动作标签。

普通增强可能会破坏这种关系。例如把左腕相机图像随机裁剪、右腕相机图像也随机裁剪，视觉上看起来没问题，但如果对应动作没有变化，模型学到的是错配关系。更糟的是，在接触丰富状态下，左手、右手、物体之间有物理约束，随便扰动一个手的 pose 可能会让另一个手的 action 标签失效。

D-CODA 的思想是做 coordinated augmentation：

```text
generate new left wrist observation
generate new right wrist observation
generate valid coordinated action label
preserve contact / bimanual consistency
```

这对数据利用率的提升来自“把一条真实双臂示教扩展成多条仍然物理合理的双臂样本”。它和普通 data augmentation 的区别是，普通增强只关心输入不变性，D-CODA 关心输入和动作标签一起变，而且变完仍然满足双臂协作约束。

对 embodied-arena 的落地建议：

- 仿真里可以先做几何级增强，不一定马上训练生成模型：

```text
object pose perturbation
camera pose perturbation
grasp point perturbation within affordance region
left/right synchronized trajectory perturbation
```

- perturb 后要重新验证：

```text
no collision violation
grasp/contact remains valid
relative hand-object constraint remains valid
task target still reachable
```

- 对双腕相机数据，要保证左右图像和动作同步变换。

建议做一个小规模实验：

```text
standard image augmentation
scene-level coordinated augmentation
contact-preserving coordinated augmentation
```

如果任务是接触丰富的，第三种通常更有意义。

## 9. AnyBimanual：从单臂 policy 迁移到双臂，不是简单混数据

论文：AnyBimanual: Transferring Unimanual Policy for General Bimanual Manipulation  
链接：https://arxiv.org/html/2412.06779v2

这类工作的重点是利用已有单臂数据或单臂 policy。它回答的问题很现实：双臂示教贵，单臂数据多，能不能复用？

直接把单臂数据混进双臂训练通常不够。原因是单臂数据没有双臂 coordination，没有另一只手造成的遮挡、接触和约束。AnyBimanual 的思路更像：

```text
已有单臂 skill/policy -> 作为可复用动作能力
skill manager -> 决定什么时候调用哪个单臂技能
compensation module -> 根据双臂场景补偿另一只手和物体约束
```

也就是说，它不是把单臂数据当成双臂数据，而是把单臂能力当成模块，然后在双臂任务里学习如何组合、调度和修正。

这很符合平台化任务设计。很多双臂任务其实由单臂技能组成：

```text
reach
grasp
pull
push
place
insert
release
```

双臂难点不是每个技能本身，而是什么时候左手执行、什么时候右手执行、什么时候两只手同时执行、什么时候一只手必须等待另一只手完成稳定。

对 embodied-arena 的落地建议：

- 先训练或复用单臂 primitive：

```text
single_arm_reach
single_arm_grasp
single_arm_place
single_arm_pull
single_arm_push
```

- 双臂策略只学习 high-level scheduling：

```text
left_skill_id
right_skill_id
skill_parameters
sync_condition
```

- 再加低层补偿：

```text
avoid collision between arms
maintain object constraint
adjust grasp pose
wait until other arm stabilizes
```

这条线的优点是非常省双臂数据，缺点是系统复杂度更高，而且需要任务能拆成已有技能。

适合：

- 家务类长时程任务。
- 多阶段 tabletop manipulation。
- 任务可以被明确分解成 primitive 的场景。

不适合：

- 高度连续耦合的双手动作，比如同步揉捏、复杂布料折叠。

## 10. MonoDuo：用“单臂机器人 + 人手”采双臂相关数据

论文：MonoDuo: Using One Robot Arm to Learn Bimanual Policies  
链接：https://arxiv.org/abs/2605.29298

MonoDuo 关注另一个现实问题：目标双臂机器人数据难采，但人可以用手和单臂机器人协作。能不能用这种数据训练双臂 policy？

它的直觉是：双臂任务最稀缺的是 coordination pattern，而不是某个机器人关节轨迹。人手和机器人协作时，已经包含了很多双臂协作信息：

```text
谁先动
谁稳定
哪里接触
物体怎么变化
两边如何同步
```

然后再把人手侧的行为 retarget 到目标机器人另一只手上。这样可以减少真正目标双臂机器人的示教数量。

这对 embodied-arena 的启发是：不要把数据来源限制在“完全同构双臂机器人演示”。可以考虑更宽的数据混合：

```text
single-arm robot demos
human-hand + robot demos
teleop bimanual demos
scripted stabilizer + learned actor demos
scripted actor + learned stabilizer demos
```

关键是训练时要把不同来源统一到任务结构上，比如统一成：

```text
left/right role
object trajectory
contact event
keypose sequence
```

而不是硬把所有数据都转成同一种 joint action。

仿真平台里可以做一个简化版：

- 用规则控制器生成 stabilizing arm。
- 只人工或策略生成 acting arm。
- 再把这类数据转成完整双臂示教。
- 和纯双臂人工示教做数据效率对比。

这个实验能回答：双臂数据里，到底是低层双臂轨迹重要，还是角色和物体运动结构更重要。

## 11. InterACT：显式建模 action chunk 内部的双臂依赖

论文：InterACT: Inter-dependency Aware Action Chunking with Hierarchical Attention Transformers for Bimanual Manipulation  
链接：https://arxiv.org/abs/2508.07898

ACT 类方法默认已经会预测未来一段 action chunk，但它未必显式区分 chunk 内部的左右臂依赖。InterACT 关注的就是这个问题：action chunk 不是一串独立动作，里面有时间依赖，也有左右臂依赖。

可以把一个双臂 action chunk 看成一个二维结构：

```text
time axis: t, t+1, ..., t+H
arm axis: left, right
```

普通 chunking 可能把它展平成一个大向量：

```text
[left_t, right_t, left_t+1, right_t+1, ...]
```

这样虽然能训练，但模型不一定知道哪些依赖更重要：

- 同一只手在不同时间之间的连续性。
- 同一时间左右手之间的同步关系。
- 左手当前动作对右手未来动作的影响。
- 右手接触后左手是否应该保持或释放。

InterACT 的启发是用层级 attention 分别建模这些关系。即使不完全复现论文，也可以把思想落到你们的模型设计里：

```text
temporal attention: 每只手自己的时间连续性
cross-arm attention: 左右手同一时间或跨时间依赖
scene attention: 手和物体/任务之间的依赖
```

这比单纯“ACT 预测双臂 action chunk”更细。它回答的是：chunk 里左右手动作怎么互相条件化。

对 embodied-arena 的实验建议：

```text
flat action chunk head
per-arm temporal head + concat
per-arm temporal head + cross-arm attention
hierarchical temporal/cross-arm attention
```

如果数据少，显式结构通常比纯大 MLP 更稳。

## 12. Diffusion-Based Imaginative Coordination：用“想象协作状态”增强双臂推理

论文：Diffusion-Based Imaginative Coordination for Bimanual Manipulation  
链接：https://arxiv.org/abs/2507.12738

这类方法关注另一个问题：双臂策略不能只根据当前观测反射式输出动作，它需要在脑子里预演“如果两只手这样动，物体和对方手会发生什么”。

所谓 imaginative coordination，可以理解为让模型生成或预测中间协作状态，再用这些状态辅助动作决策。它和 future state modeling 有点像，但更强调双臂之间的协同过程，而不是只预测最终物体状态。

人话解释就是：

```text
不要只问：现在该输出什么 action？
还要问：左手这样动以后，右手需要配合到什么状态？
物体会变成什么状态？
下一阶段接触关系会是什么？
```

这对数据利用率的帮助在于，模型不只是记住示教 action，而是在学习任务中间状态的演化。中间状态通常比动作更稳定，也更容易跨场景迁移。

在 embodied-arena 里可以用更可控的方式实现类似思想：

```text
predict next contact mode
predict next left/right keypose
predict next object pose
predict whether both hands should synchronize
```

这些预测可以作为 auxiliary loss，也可以作为 planner/policy 的中间变量。

## 13. 应该怎么设计你的对比实验

如果你的目标是研究“单臂 vs 双臂训练到底不一样在哪里”，建议不要只比较单臂和双臂最终成功率。更好的实验矩阵是逐步加结构。

最低限度：

```text
A. single-arm policy on single-arm tasks
B. bimanual concat BC
C. bimanual joint policy with shared encoder
D. C + left/right arm tokens + cross-attention
E. D + relative relation state
F. E + role decomposition
G. E + keypose/object future auxiliary loss
H. E + coordinated data augmentation
```

每个实验回答的问题不同：

```text
B: 只扩 action/state 是否足够
C: 共享感知是否能提高样本效率
D: 显式左右臂交互是否重要
E: hand-object/hand-hand 相对关系是否重要
F: support-act 结构是否能减少示教需求
G: 中间监督是否比 dense action 更稳
H: 双臂一致性增强是否比普通增强有效
```

建议按数据量画曲线，而不是只报一个数据点：

```text
10 demos
20 demos
50 demos
100 demos
full demos
```

数据利用率的关键不是满数据下谁最高，而是少数据区域谁先起来。

## 14. 适合 embodied-arena 的 observation/action 设计

推荐 observation 不只包含绝对 state：

```text
left_ee_pose
right_ee_pose
object_pose
left_gripper_state
right_gripper_state
object_joint_state
```

还要包含相对关系：

```text
left_ee_pose_in_object_frame
right_ee_pose_in_object_frame
right_ee_pose_in_left_ee_frame
left_contact_flag
right_contact_flag
left_contact_point_in_object_frame
right_contact_point_in_object_frame
```

如果任务有 articulated object：

```text
target_joint_name
current_joint_pos
target_joint_pos
handle_pose
left_ee_pose_in_handle_frame
right_ee_pose_in_handle_frame
```

如果任务有 role：

```text
left_role_token
right_role_token
stabilizing_region
acting_region
```

action 可以从简单到复杂逐步做：

```text
baseline: [left_delta_ee_pose, left_gripper, right_delta_ee_pose, right_gripper]
role-based: [stabilizer_target, actor_delta_action]
keypose-based: [left_keypose, right_keypose, local_controller_params]
skill-based: [left_skill, left_skill_params, right_skill, right_skill_params]
```

注意项目的 Isaac Lab 四元数规范：姿态里的 quaternion 必须是 `[x, y, z, w]`，不要混用旧的 `[w, x, y, z]`。

## 15. 方法选择建议

如果你的任务是开盖、开抽屉、插拔、切割、清洁：

```text
优先做 role decomposition
加 acting/stabilizing affordance
加 hand-object relative pose
```

如果你的任务是双手搬运、托盘、长物体移动：

```text
优先做 triadic relation
加 future object pose prediction
加 hand-hand relative pose
```

如果你的任务是布料、绳子、柔性物体：

```text
优先做 object keypoint/pointflow
加 future state auxiliary loss
必要时做阶段分解
```

如果你的目标是最大化利用已有单臂数据：

```text
优先做 skill transfer
单臂 primitive 预训练
双臂 high-level scheduler
少量双臂数据 finetune compensation
```

如果你的数据来自双腕相机：

```text
优先看 coordinated data augmentation
不要只做普通 image augmentation
保证左右图像和动作标签一致变化
```

## 16. 最推荐优先实现的三个实验

如果只能先做三个，我建议：

### 实验一：relative relation state

比较：

```text
absolute state only
absolute + hand-object relation
absolute + hand-object + hand-hand relation
```

这是最容易实现、最容易解释、和双臂本质最相关的实验。

### 实验二：acting/stabilizing role decomposition

选 support-act 任务，比如开盖、开抽屉、插拔。比较：

```text
joint BC
joint BC + role token
stabilizer pose head + actor action head
```

这个实验能明显展示“双臂结构先验”对数据利用率的帮助。

### 实验三：keypose/object future auxiliary loss

比较：

```text
action BC only
action BC + keypose prediction
action BC + future object state prediction
action BC + both
```

这个实验能验证模型是不是因为理解了任务中间状态而提升，而不是只拟合动作。
