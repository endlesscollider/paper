# 双臂任务中 action / trajectory 扰动如何提升闭环成功率：论文调研

资料日期：2026-06-17

本文只讨论你关心的部分：**不谈纹理、背景、桌面、光照这类视觉随机化，重点看 action、trajectory、waypoint、relabel、双臂协调约束，怎么让闭环成功率变高**。我把近两年比较相关的双臂论文按工程可落地程度来讲。每篇都尽量说人话：它在解决什么问题，为什么这个方法能泛化，里面的 action 到底怎么表示，扰动/合成数据怎么做，如果要在 embodied-arena / Isaac Lab 里复现，大概应该怎么落。

参考论文与项目：

- D-CODA: Diffusion for Coordinated Dual-Arm Data Augmentation, 2025, <https://arxiv.org/html/2505.04860v2>
- GLIDE: Planning-Guided Diffusion Policy Learning for Generalizable Contact-Rich Bimanual Manipulation, 2024/2025, <https://arxiv.org/html/2412.02676v2>
- BiDemoSyn: One-Shot Real-World Demonstration Synthesis for Scalable Bimanual Manipulation, 2026, <https://arxiv.org/html/2512.09297v3>
- RDT-1B: a Diffusion Foundation Model for Bimanual Manipulation, 2024, <https://arxiv.org/abs/2410.07864>
- AnyBimanual: Transferring Unimanual Policy for General Bimanual Manipulation, 2024/2025, <https://arxiv.org/html/2412.06779v2>
- ACT / ALOHA: Learning Fine-Grained Bimanual Manipulation with Low-Cost Hardware, RSS 2023, <https://arxiv.org/abs/2304.13705>

## 先给结论：action 扰动不是给 label 加噪声

对双臂 manipulation，尤其是 lift、handover、push、insert、unscrew、pour、reorient 这种任务，闭环失败的根源通常不是模型没见过某个背景，而是**执行时稍微偏离专家轨迹后，不知道怎么恢复**。行为克隆训练的是 `obs_t -> expert_action_t`，但真实 rollout 时 policy 会犯小错，小错会把状态带到训练集没覆盖的位置，下一步又更错，最后抓偏、掉物、顶住、夹空、两臂不同步。

所以 action 扰动的正确目标不是“让动作更随机”，而是构造这类样本：

```text
原本专家轨迹附近的状态
-> 执行一个合理的小偏差
-> 得到一个偏离但仍可恢复的状态
-> 给这个偏离状态重新标注 recovery action / recovery chunk
-> 训练 policy 学会回到任务轨道
```

错误目标是：

```text
原始 obs_t -> expert_action_t + random_noise
```

这个通常只是把监督标签污染了。尤其双臂里，两个臂的动作不是独立的。比如双臂托球，左臂向左扰了 2cm，右臂如果不跟着保持相对距离和姿态，球就掉；双臂拧瓶盖，左臂固定瓶身，右臂旋转瓶盖，两个动作的相位、接触力方向、相对位姿都有约束。你不能简单给左右臂各自采一个独立高斯噪声。

更好的扰动应该满足四个条件：

1. **小幅**：扰到 policy 真会遇到的错误附近，不要扰到任务不可恢复。
2. **低频**：扰动一个 waypoint、一个 action chunk 或一段轨迹，不要每个控制步白噪声乱抖。
3. **phase-aware**：不同阶段扰不同东西。接近阶段扰位姿，抓取阶段扰抓点和夹爪时机，接触阶段扰双臂相对约束，放置/插入阶段扰目标前 waypoint。
4. **relabel**：扰动后的状态必须有新的正确动作标签。这个标签可以来自 IK、planner、scripted oracle、human intervention、原轨迹对齐后的未来 waypoint，或者 contact-aware optimization。

下面几篇论文的共同点就是：它们都没有把“双臂 action 扰动”当成简单噪声，而是围绕**可恢复状态、物理可行性、双臂协调关系、动作表示**来设计。

## 1. D-CODA：最贴近“action 扰动 + 双臂约束 + relabel”的论文

D-CODA 是这几篇里最直接回答你问题的一篇。它的全名是 Diffusion for Coordinated Dual-Arm Data Augmentation。虽然论文里有 wrist camera image synthesis，但真正值得我们看的是它对**双臂动作标签怎么跟着扰动一起变**的处理。它解决的问题很具体：双臂 eye-in-hand imitation learning 里，真实 demo 很少，policy 一旦闭环 rollout 偏离原始 wrist-camera 视角，就会出错。单臂里已经有人做过 novel-view augmentation 和 action label relabel，但双臂不能直接照搬，因为左右臂看到的是同一个任务的两个局部视角，两个视角必须一致，动作也必须满足双臂协调。

论文的任务包括仿真和真实双臂任务。真实平台是两台 UR5，每个臂有 Robotiq 2F-85 夹爪和 RealSense D415 wrist camera。真实任务有 Lift Ball、Lift Drawer、Push Block。仿真任务里也有 Coordinated Lift Tray、Dual Push Buttons、Bimanual Straighten Rope 这类需要协同或同时操作的任务。下游 policy 主要用 ACT，也试了 VLA 路线的 pi0-FAST 微调。它的结论不是“生成图像就一定好”，而是：当 augmentation 能生成**偏离原轨迹但动作标签仍然合理的双臂样本**时，闭环失败会减少。论文里还观察到 D-CODA 和 bimanual DMD 这种会制造 OOD state 的方法，policy 能学到某些 recovery 行为，比如 gripper 滑离物体后重新定位继续推。

它的方法可以拆成三件事。

第一，判断当前状态是 contactless 还是 contact-rich。论文用 SAM2 分割 gripper，再结合 depth 或 SSIM 判断 gripper-to-object contact。这个判断很重要，因为双臂扰动的自由度在两个阶段完全不同。没有接触时，左右臂可以更独立地扰动，因为动作只是“从当前位置去往某个接近目标”；接触后，左右臂和物体形成了一个小系统，两个末端之间的相对关系往往就是成功条件。比如两个夹爪托住球，左右臂之间距离太大，球掉；距离太小，挤压过大；一边抬一边不抬，物体倾斜。所以 contact-rich 阶段不能各自随机扰。

第二，采样 pose perturbation。contactless 阶段，D-CODA 对每只手的 wrist camera / end-effector pose 在范围内均匀采样平移和旋转扰动。论文给的实现细节里，平移扰动量级是 0.01 到 0.02 米，contactless 的旋转范围大约是正负 28.7 度。这个量级很有启发：它不是大范围重规划，而是让 policy 学会处理专家轨迹附近的小偏差。contact-rich 阶段，它不随机采两个独立扰动，而是做 constrained optimization。核心直觉是：接触时对两只手施加相同或协调的扰动，保持必要的相对距离、相对姿态和可达性。优化变量是扰动平移方向，约束包括不要离桌子太近、两个 end-effector 不要太近、IK 必须可解。这个设计就是双臂 action augmentation 的重点：扰动要让状态变新，但不能把任务结构破坏掉。

第三，生成新的 action label。D-CODA 不是把原动作原封不动贴到新图像上。它用 camera perturbation、原 camera pose、原 end-effector pose 做坐标变换，得到扰动后的 end-effector target pose，然后用 Levenberg-Marquardt IK 算出新的目标 joint positions。IK 不可解就丢弃这个增强样本。下游 policy 学的是 joint-space target actions。它还不是每一帧都替换，而是每隔 `k` 个 timestep 替换一个原状态，论文里常用 `k=6` 或 `k=9`。这个细节也很重要：如果你把整条轨迹每一步都替换成强扰动，训练分布会变得不连续，policy 可能学到很怪的跳跃动作；间隔替换相当于在原轨迹里插入 recovery 点，让 policy 学到“偏了之后怎么回到主轨道”，同时保留足够多原始 in-distribution 动作帮助完成任务。

为什么 D-CODA 能提高闭环成功率？用人话说，它给训练集补了三类原始 demo 里少见但测试时常见的数据。第一类是视角和末端位姿略偏的状态。第二类是双臂接触约束还没坏、但已经偏离理想轨迹的状态。第三类是这些状态对应的恢复动作。普通 BC 只看完美专家轨迹，遇到稍微偏离就懵；D-CODA 让 policy 在训练时见过“偏一点但还能救”的情况。对于双臂，这种数据尤其值钱，因为双臂失败常常不是单臂绝对位置错，而是**左右臂相对关系错**。

如果我们要在 embodied-arena 里做一个 D-CODA 精简版，我建议先不要做 diffusion 图像生成，先做 action / state 层面的版本。因为你现在关注的是 action perturbation，而且仿真里能直接拿到状态，不需要先解决 novel wrist image synthesis。

具体步骤可以这样：

1. 给每条双臂 demo 做 phase 标注或自动检测。最少分成 `approach`、`pre_contact`、`contact_transport`、`release/retract`。如果任务有插入/拧/倒水，再加 `align`、`insert`、`twist`、`pour`。自动检测可以先用 gripper distance、object contact pair、object velocity、gripper open/close event，不必上来就做复杂模型。
2. 定义每个 phase 的扰动空间。`approach` 可以扰左右 TCP 的 xyz 和 yaw；`pre_contact` 扰小一点；`contact_transport` 改成扰双臂公共刚体变换，或者扰 object-centric frame 下的共同平移/旋转；`insert/twist` 扰动量更小，主要扰目标前 waypoint 和时间相位。
3. 对 contactless phase，左右臂可以独立采样，但要检查 IK、碰撞、关节限位、双臂互碰。对 contact-rich phase，先从 object frame 或 dual-arm frame 采样一个共同扰动，再把它作用到左右 TCP，保持左右 TCP 相对距离和关键方向。
4. 扰动后不要直接保存 noisy action。要把仿真状态推进到扰动后的状态，或者直接构造扰动状态，然后用 planner / IK / task oracle 生成 recovery chunk。比如抓偏后重新回到 pre-grasp，托物偏了后让双臂共同调整到稳定相对位姿，插入偏了后退一点再重新 align。
5. 对每个增强样本保存 `(obs_perturbed, action_recovery_chunk, phase, perturb_meta, valid_flag)`。其中 `perturb_meta` 很有用，后面可以分析哪类扰动真正提升成功率。
6. 训练时不要让增强样本淹没原数据。可以从 `original:augmented = 1:1` 或 `2:1` 开始，recovery 样本 loss 权重给 2 到 5 倍。关键不是数量越多越好，而是临界失败附近的数据质量。
7. 评测时要单独做 perturb eval。比如初始位姿不变，但 rollout 中在 pre-grasp 人为把物体推 1cm、把一个臂延迟 2 step、让两个 TCP 相对距离偏 1cm，看 policy 能不能救回来。只看 clean success rate 会掩盖 recovery 能力。

这篇对我们最大的启发是：**双臂 action 扰动必须分 contactless 和 contact-rich；contact-rich 下扰动的是双臂系统，不是两个独立 action**。如果一个方法没有这个意识，那它大概率只能在简单 pick/place 或视觉变化上有效，一到托举、插入、双臂推、双臂旋转就会产生物理不合理样本。

## 2. GLIDE：用 planner 生成大规模 contact-rich 双臂轨迹，再训练 diffusion policy

GLIDE 不是一篇专门讲“给 action 加扰动”的论文，但它对你的问题很重要，因为它回答了另一个核心点：**什么样的 action 表示和轨迹数据更容易在双臂 contact-rich 任务中泛化**。D-CODA 更像是在已有真实 demo 周围生成恢复样本；GLIDE 更像是从任务物理出发，用 contact-rich planner 在仿真里生成大量高质量轨迹，再训练一个闭环 policy 去模仿这些轨迹。它的目标任务是双臂重定位/重定向桌面上的大物体。物体可能很重、很大，不能像普通抓取那样夹起来，只能用两只机械臂和物体多处接触，把物体推、顶、转到目标姿态。

这类任务和常见双臂 pick-up 不一样。双臂不是两个夹爪一左一右夹住然后抬起来，而是整个手臂和物体形成多接触系统。动作失败常见于接触点选错、两臂力方向不配合、动作不光滑、一个臂先顶走物体另一个臂没跟上。GLIDE 的做法是先用 contact-implicit trajectory optimization / smoothed contact model 这样的 planner 在高保真仿真里生成 demo。planner 有完整 object pose 和几何信息，可以慢慢算；训练好的 policy 部署时只用点云观察和目标任务条件，不需要在线知道完整物体模型，也不需要实时跑慢 planner。

GLIDE 的数据生成思路很工程化。它在仿真里随机生成 2000 个 rectangular box primitives，随机尺寸、质量、摩擦、初始 pose 和目标 pose。然后用 planner 生成轨迹，再在高保真 simulator 里 rollout 验证。没达到目标或太慢的轨迹丢掉，只保留高质量 demo。最后保留 12000 条成功轨迹训练 diffusion policy。这里的重点不是“随机化物体外观”，而是扰动和覆盖**动作需要适应的物理因素**：物体尺寸、质量、摩擦、初始姿态、目标姿态。这些因素直接改变双臂应该怎样接触、推、转、绕行。换句话说，GLIDE 的泛化来自任务动力学空间覆盖，而不是图像花样。

它的 action 设计非常值得注意。论文明确说，直接预测 absolute end-effector poses 或 absolute joint angles 会导致 unseen objects 上泛化差、真实轨迹不光滑。GLIDE 改成预测 residual joint position action：

$$
a_{t+1:t+T} = { q_i - q_t }
$$

也就是以当前关节状态为基准，预测未来一段 joint position 的增量序列。这个设计有两个好处。第一，不同轨迹、不同物体、不同初始姿态下，绝对关节位置差异很大，但“下一段应该怎么动”的局部增量分布更稳定。第二，闭环执行时 residual action 不容易出现突然跳到远处配置的动作，轨迹更平滑。论文的消融很强：在固定 45 度旋转任务上，带 residual action 和 flying point augmentation 时真实成功率 0.800；去掉 residual action 后真实成功率掉到 0.520；去掉 flying point augmentation 掉到 0.320；两个都去掉真实成功率为 0。虽然 flying point 是点云噪声，不是你现在关注的 action，但 residual action 对闭环动作泛化的贡献非常直接。

GLIDE 还有一个对 action chunk 很有意义的结论。它的 diffusion policy 预测一段 action sequence，训练时 horizon 是 64，测试时发现执行 horizon `T_a=20` 更好。`T_a=8` 太短，轨迹不够平滑，成功率低；`T_a=40` 或 `64` 在随机旋转任务上反而下降，因为长 chunk 会减少实时反馈频率，policy 来不及根据物体实际运动修正。这个结论对双臂闭环非常实用：action chunk 不是越长越好。太短会抖，太长会开环。对接触任务，应该找一个平衡：chunk 足够长以产生平滑协调动作，但每隔一小段必须重新观察和重规划。

GLIDE 的泛化评测也很有价值。它训练只用 box primitives，但真实评测包括 in-distribution 盒子、OOD containers、overfilled containers、甚至 inflatable toys。这些 OOD 物体有不同几何、边界、重量和柔性。成功率会下降，但仍能完成一部分，说明 policy 学到的不是某个固定盒子的轨迹，而是“从点云边界判断物体，双臂通过接触把物体推/转到目标”的局部控制规律。这种泛化和 action 扰动有关，因为训练数据中大量随机目标姿态和物体参数迫使 policy 学局部反馈，而不是背某条轨迹。

如果我们要借鉴 GLIDE，在 embodied-arena 里做双臂任务的 action 泛化，可以分两条线。

第一条线是 planner-guided demo generation。对每个任务，先写一个不要求实时的 oracle 或 planner，哪怕很慢也没关系。比如 stack can、open laptop、open drawer、insert cans、put screws 这些任务，离线 planner 可以知道 object pose、joint state、affordance frame。它输出双臂 keypose 或 joint trajectory。然后在仿真里 rollout 验证，只保留成功轨迹。不要把 planner 生成的所有轨迹都拿来训练，因为接触 planner 经常会生成“理论上能走但实际 rollout 差一点失败”的样本，这种样本会污染 policy。

第二条线是 residual / object-centric action representation。我们可以比较几种 action：

```text
absolute joint target
joint delta / residual joint target
absolute TCP pose
TCP delta in robot base frame
TCP delta in object frame
dual-arm relative frame action
```

对双臂泛化，我会优先试 `object-centric TCP delta + gripper` 或 `residual joint target + phase-conditioned policy`。如果策略最终控制的是 humanoid/dual-arm，不同 embodiment 之间 joint 不同，那么 object-centric TCP delta 更有跨本体意义；如果固定机器人，residual joint target 往往更稳。GLIDE 的实验证明：在真实 contact-rich 双臂任务里，residual 比 absolute 更可靠。

具体实现建议：

1. 先给任务定义 object frame 和 goal frame。比如杯子、罐子、抽屉把手、笔帽、瓶盖、容器边缘，都要有可查询的 frame。
2. 轨迹生成时保存每个 timestep 的 `q_t`、left/right TCP pose、object pose、contact state、goal pose。
3. 训练 label 同时导出多种版本：`joint_abs`、`joint_delta`、`tcp_delta_base`、`tcp_delta_object`。不要一开始只保存一种，后面很难比较。
4. 对 action chunk 做 sweep：8、16、20、32、64。评估不仅看 clean success，还看中途扰动恢复和轨迹平滑度。
5. 对 planner 数据做 filter。成功、时间短、接触稳定、没有大碰撞、动作平滑的轨迹保留；接近目标但动作暴力的不要。
6. 对 contact-rich phase 增加局部扰动：从成功轨迹中取状态，扰 object pose 或 TCP pose 小量，然后让 planner 从该状态重新生成一小段 recovery chunk。这就是 GLIDE 和 D-CODA 的结合。

GLIDE 给我们最大的启发是：**闭环泛化不只来自“见过偏离状态”，也来自 action 表示把不必要的绝对量去掉**。如果 label 本身是 absolute joint pose，模型学到的是很多和初始配置绑定的东西；如果 label 是 residual 或 object-centric delta，模型更容易学到局部纠偏规律。

## 3. BiDemoSyn：从一条真实双臂 demo 合成大量物理可行轨迹

BiDemoSyn 是很适合工程团队参考的一篇，因为它不是从“训练一个更大的模型”出发，而是从数据生产出发：双臂 demo 采集太贵，仿真数据又有 sim-to-real gap，那能不能只采一条真实双臂示教，然后把它变成很多条仍然物理可行的真实轨迹？它的答案是：可以，但不能简单平移整条轨迹，也不能随便扰动每个 waypoint。必须把任务拆成**不变的双臂协调块**和**随物体/场景变化的可调整块**。

论文的设置是固定双臂平台，两只 6-DoF AUBO-i5 协作臂，平行夹爪，第三视角双目相机。任务包括 plugpen、inserting、unscrew、pouring、pressing、reorient，后面还扩展了 reorient+unscrew、unscrew+pouring 这种更长任务。它用 kinesthetic teaching 采一条示教，记录 6-DoF end-effector poses 和 gripper binary states。然后通过三阶段生成大量 synthetic demonstrations：一是分解 one-shot teaching，二是视觉对齐新初始状态，三是轨迹调制和优化。

它最值得看的概念是 invariant coordination blocks 和 variable blocks。双臂任务里有些动作结构基本不变，比如双臂同时托起、左右手保持相对位姿旋转、一个手稳定瓶身另一个手拧、交接物体时的同步关系。这些块不能被随意扰动，否则任务语义就没了。另一些动作是和物体初始位置、姿态、形状相关的，比如去哪里抓、从哪个方向接近、目标前 waypoint 在哪。这些块应该根据新场景改。BiDemoSyn 的核心就是：**只扰动应该随场景变化的部分，保留任务语义和双臂协调结构**。

这和我们讨论的 action 扰动高度相关。很多人做数据增强时会犯一个错：把整条轨迹套一个 SE(3) 变换，或者对每个 waypoint 加随机噪声。对单臂 pick/place 也许还能用；对双臂任务很危险。比如 unscrew，左臂固定瓶身，右臂旋盖，右臂绕瓶盖轴旋转是关键语义；如果你对右臂每个 waypoint 加独立噪声，旋转轴就歪了，轨迹不再是拧。再比如 pouring，倾倒角度、杯口和瓶口相对位置、水流方向是关键；简单平移可以，随机扰动姿态可能会把 pouring 变成洒水。BiDemoSyn 的做法是 block-level modulation，不是 point-wise noise。

论文里还有一个比较具体的双臂协调建模：它的 diffusion / trajectory model 使用带左右臂相关项的协方差结构，里面有左臂时间平滑、右臂时间平滑、左右臂依赖和 coordination strength。你不一定要照抄公式，但工程含义很清楚：生成双臂轨迹时，左右臂不应该是两个独立序列。它们的噪声、平滑、同步和相对关系都应该被建模。最朴素的实现就是：在 `contact_transport`、`handover`、`twist` 这类 block 里，采样一个 shared latent perturbation，再采很小的 per-arm residual perturbation；不要给两只手完全独立采大噪声。

BiDemoSyn 的轨迹可行性检查也很实用。每个 adapted block 都要做 IK reachability 检查、关节限位检查、双臂/场景碰撞检查、两点 motion planning 检查。不可达或奇异的样本删除。论文补充材料里说超过 96% 的轨迹能通过验证。这个比例背后有个工程经验：扰动范围要围绕真实示教的可达区域设计，而不是让 optimizer 去救所有随机样本。先把样本采在合理区域，再用 IK/planner 过滤，效率和质量都更好。

它的动作表示是 object-centric observation + 6-DoF end-effector action。这个点对跨 embodiment 很关键。论文把在主双臂平台上训练的 policy zero-shot 部署到另一个 humanoid-style 双臂平台，只做坐标变换，不重新训练。能做到这一点，靠的是 policy 输出的不是源机器人 joint，而是更接近任务语义的 6-DoF 末端动作。不同机器人只要能用 IK 执行这些末端动作，就能迁移。对 embodied-arena 如果有 dual RM、humanoid arms 或其他双臂配置，joint-level policy 可能会被本体绑定；TCP-level object-centric policy 更适合作为泛化实验的 action space。

BiDemoSyn 的实验结论也支持“扰动覆盖越合理，泛化越好”。它做了空间覆盖和朝向多样性的分析。比如 unscrew 中，训练数据的 workspace coverage 从 sparse 到 dense，提高了 unseen position 的成功率；reorient 中，orientation diversity 从低到高，成功率明显提升。这里说的 diversity 不是视觉背景，而是 action 必须适应的 object pose / orientation diversity。对双臂而言，这种 diversity 直接决定 policy 是否学会“相同协调结构在不同物体位姿下怎么落地”。

如果我们要把 BiDemoSyn 思路落到 embodied-arena，我建议做一个“block-based trajectory augmentation”模块。

第一步，定义统一的双臂轨迹格式：

```text
episode
  observations
  left_tcp_pose_xyzw
  right_tcp_pose_xyzw
  left_gripper
  right_gripper
  object_poses
  phase / block_id
  contact_state
  action_label
```

注意四元数顺序必须用 Isaac Lab 新版 `[x, y, z, w]`。

第二步，做 block segmentation。可以先人工或规则化，不必追求自动完美。比如：

```text
reach_left_object
reach_right_object
dual_pregrasp_align
grasp_or_contact
coordinated_transport
twist_or_insert_or_pour
release
retract
```

每个 block 标注类型：`invariant` 或 `variable`。`variable` 块可以根据新 object pose 重新对齐；`invariant` 块只允许整体刚体变换或很小扰动，保持左右臂相对约束。

第三步，对 variable block 做 object-frame retargeting。例如原示教里左手相对杯子的 pre-grasp pose 是 `T_object^-1 * T_left_tcp`，新场景杯子 pose 变了，就用 `T_object_new * relative_pose` 得到新 TCP target。右手同理。如果是双臂共同抓同一物体，左右手 relative pose 应该同时由同一个 object frame 推出来，而不是分别采。

第四步，对 invariant block 做 coordination-preserving perturbation。比如双臂托举时，保存原轨迹中 `T_left^-1 * T_right` 或 object frame 下左右 TCP 的相对 pose。新轨迹可以整体平移/旋转，但左右臂相对 pose 不能乱。对 unscrew，可以固定瓶身手的稳定 pose，同时右手绕目标轴旋转，扰动的是轴位置和初始相位，不是整个旋转序列独立噪声。

第五步，所有新 keypose 都经过 IK、碰撞、关节限位、轨迹平滑检查。失败样本删掉，不要硬塞给模型。

第六步，生成 action label。可以是 6-DoF TCP delta，也可以转成 joint delta。若目标是跨 embodiment，优先保存 TCP delta in object frame；若目标是固定机器人闭环成功率，joint delta 也可以保留。最好两种都保存，以便对照。

第七步，训练和评估。训练时把原始 demo、block-retarget demo、局部 recovery demo 混合。评估时用 held-out object pose、held-out orientation、held-out object instance、mid-rollout perturbation 四类 split。尤其要测“执行到一半物体被推偏 1cm / 旋转 5 度后能不能恢复”，这是 block-based augmentation 的强项。

BiDemoSyn 给我们的最大启发是：**好扰动不是对轨迹点加噪声，而是先理解任务里哪些结构不能变，再只扰动应该变的部分**。双臂任务的泛化，本质上是“保持协调语义，适配新的物体状态”。

## 4. RDT-1B：大规模双臂 diffusion foundation model，看的是统一动作空间和多模态动作分布

RDT-1B 和前面三篇不太一样。D-CODA、GLIDE、BiDemoSyn 更像数据增强/轨迹生成方法；RDT-1B 是大模型路线，目标是训练一个 1B 级 Robotics Diffusion Transformer，用于双臂 manipulation。它强调两个难点：双臂任务动作维度高、动作分布多模态；同时高质量双臂训练数据稀缺。为了解决这两个问题，它用 diffusion model 表示复杂动作分布，并提出 physically interpretable unified action space，试图把不同机器人、不同任务的数据合到一个可训练的动作空间里。

这篇对你关心的 action 扰动没有直接给出“怎么扰”的配方，但它对“什么 action 表示能泛化”很有价值。大模型能不能跨任务、跨场景、跨机器人泛化，核心不只是模型参数大，而是动作空间要统一。如果不同数据源里，有的用 joint target，有的用 TCP pose，有的用 gripper command，有的控制频率不同，有的左/右臂顺序不同，直接混起来会很难学。RDT 的思路是把动作变成物理可解释的统一表示，让模型看到的 action token / action vector 尽量表达相同含义。

对双臂而言，“物理可解释”至少包括三个层次。第一，左右臂要有明确语义，不是简单拼接一堆数字。模型要知道哪一部分是 left arm，哪一部分是 right arm，哪一部分是 gripper。第二，动作应该接近执行效果，而不是过度绑定某个机器人关节编号。比如末端位姿变化、夹爪开合、相对物体的运动，比某个机器人第 3 关节加 0.1 更可迁移。第三，动作序列要能表达多模态。双臂任务常有多种成功路径：从左侧绕、从右侧绕、先用左手稳定再右手操作、或两臂同时接近。L2 回归容易把多种动作平均成一个无效动作，diffusion policy 更适合表示这种多峰分布。

RDT 对闭环成功率的启发主要有三点。

第一，双臂 action 最好不要只做单步回归。单步动作很容易抖，而且不能表达一段协调动作的相位关系。ACT、Diffusion Policy、GLIDE、RDT 都在用 action sequence / chunk，只是模型形式不同。RDT 的 diffusion transformer 更强调在大规模数据上学习动作序列分布。对我们的工程实现来说，即使不用 RDT，也应该让 policy 输出 `left/right action chunk`，并通过 receding horizon 执行前几步再重新观察。

第二，统一动作空间本身就是一种泛化手段。如果你希望同一个模型覆盖多个双臂任务，甚至多个机器人，那么数据导出阶段必须标准化 action。不要今天一个任务存 joint target，明天一个任务存 TCP delta，后天一个任务存 planner waypoint。建议至少统一成：

```text
left_tcp_delta_position
left_tcp_delta_rotation_6d 或 quaternion_xyzw_delta
left_gripper_command
right_tcp_delta_position
right_tcp_delta_rotation_6d 或 quaternion_xyzw_delta
right_gripper_command
optional: phase_id / task_id / object_frame_id
```

如果实际控制器需要 joint，就在部署时用 IK / controller 转。训练数据里仍然可以附带 joint 信息，但 policy 主 action 最好是可解释的 TCP/object-centric 表示。

第三，大规模数据并不替代 recovery 数据。RDT 这类模型能从大数据学到很多先验，但如果数据主要是成功专家轨迹，闭环偏离后的状态仍然不足。对我们的任务，最合理的路线不是“等一个 RDT 解决所有问题”，而是用 RDT 思路规范动作空间，然后用 D-CODA / BiDemoSyn / GLIDE 的思路生成 recovery 和扰动轨迹。换句话说，foundation model 提供强 prior，action perturbation 提供目标任务闭环鲁棒性。

如果要把 RDT 的思想落地到 embodied-arena，不一定要训练 1B 模型。可以先做一个“RDT-lite 数据标准和模型接口”。

第一步，统一导出双臂 action schema：

```text
timestamp
task_name
phase
left_tcp_pose_world
right_tcp_pose_world
left_tcp_pose_object
right_tcp_pose_object
left_action_tcp_delta_object
right_action_tcp_delta_object
left_gripper
right_gripper
joint_pos
joint_vel
object_pose
success_label
recovery_label
```

这里姿态可以考虑 rotation 6D，减少 quaternion 符号翻转问题；如果用 quaternion，严格使用 `[x, y, z, w]`。

第二步，训练 action chunk 模型，而不是单步模型。模型输入可以是图像/点云/状态，输出未来 `H=16/20/32` 步双臂 TCP delta。执行时只执行前 `E=4/5/8` 步，再重新观察。这样同时保留平滑性和闭环反馈。

第三步，显式建模左右臂。不要把 14 维 action 当普通向量随便 MLP 输出。至少在网络里保留 left token、right token、object token、task token，让模型学相互 attention。双臂协调失败往往来自左右臂互相看不见对方意图。

第四步，数据增强时保持 unified action semantics。比如对一个双臂托举样本做扰动，不能只改 left action；如果输出是 object-frame TCP delta，就要更新左右 action 在同一 object frame 下的一致关系。

第五步，评估 cross-task / cross-object / cross-robot。RDT 的价值不在单任务 95% 成功，而在 few-shot 新任务、unseen objects/scenes、不同双臂平台上是否还能用。我们可以先用同一个 action schema 在 dual RM 和 humanoid/sharpa 任务间做迁移实验。

RDT 给我们的最大启发是：**要想双臂泛化，数据层的 action 表示必须先统一且物理有意义**。如果 action schema 混乱，再多扰动也只是把混乱放大。D-CODA 解决“偏离状态怎么恢复”，GLIDE 解决“contact-rich 轨迹怎么规模化”，BiDemoSyn 解决“少量真实 demo 怎么合成可行轨迹”，RDT 则提醒我们：这些数据最后要进入一个统一动作空间，模型才可能吃得下。

## 5. AnyBimanual：把单臂泛化能力迁移到双臂，关键是 skill scheduling 和双臂动作合理性

AnyBimanual 关注的问题和前面几篇又不一样。它的出发点是：单臂 policy 近几年数据多、模型大、泛化能力已经不错；双臂 demo 很贵，能不能把预训练单臂 policy 的能力迁移到双臂任务上？它提出一个 plug-and-play 框架，把 pretrained unimanual policy 变成 bimanual manipulation policy，只需要少量双臂 demo。论文在 RLBench2 的 12 个仿真双臂任务和 9 个真实任务上评估，真实任务平均成功率报告为 84.62%。真实平台是两台 UR5e 和 Robotiq 2F-85 夹爪。

它不是 action perturbation 论文，但里面有两个点和你关心的 action 泛化很相关。第一，双臂任务经常可以看成单臂技能的同步或异步组合。比如一只手 pick，另一只手 place；一只手 stabilize，另一只手 operate；两只手同时 lift；一只手 handover，另一只手 receive。AnyBimanual 用 skill manager 从预训练单臂 policy 里挖 skill primitives，再为左右臂动态调度这些 skill。第二，它在数据增强里提到 SE(3) augmentation，并且特别说要修改扰动策略，确保 perturb 后的双臂 ground-truth actions 仍然 reasonable。这句话虽然短，但非常关键：双臂 SE(3) augmentation 不能像单臂那样独立扰，因为左右臂动作要共同完成同一个任务。

AnyBimanual 的 action 定义是每个 end-effector 的 location、orientation、gripper open state，以及是否使用 collision avoidance in motion planner。底层执行用 low-level planner，比如 RRT-Connect 或 MoveIt，policy 输出关键末端动作。真实数据采集时，他们用两个 Xbox joystick 控制双臂，采完完整轨迹后人工提取 keyframes，减少学习难度。测试时，policy 给出两只手的 next best poses，然后 MoveIt 执行，再重新观察，再输出下一步。这其实是一种 keypose-level closed-loop policy，而不是高频 joint servo policy。

这对我们的启发是：如果双臂任务较长、动作维度高、demo 少，不一定要让 policy 每 20ms 输出连续 action。可以让 policy 输出下一组双臂 keypose，再交给 planner / controller 执行。这样 action 扰动也应该发生在 keypose 层：扰目标 keypose、扰 keypose 时序、扰左右臂同步关系，然后用 planner 验证可行，而不是在低层控制命令上加白噪声。

AnyBimanual 的 skill manager 对泛化的意义是：它避免为每个双臂任务从零学一个巨大动作空间。单臂预训练 policy 已经知道很多 primitive，比如 open drawer、push button、insert peg、stack blocks、place cups。双臂任务的难点是怎么给左/右臂分配这些 primitive，并在时间上协调。比如 push box 到目标区域，普通 PerAct2 可能只模仿一个向前推的动作，没真正让两臂都接触箱子；AnyBimanual 的 skill scheduling 能让两个臂都找到接触并协同推。这个和 action 扰动的关系在于：当你做扰动数据时，也应该保留 skill 语义和左右臂角色，不要把“稳定臂”和“操作臂”的语义扰乱。

论文也诚实地说了限制。AnyBimanual 不擅长 zero-shot unseen tasks，因为 skill manager 和 visual aligner 还是需要双臂 demo 来学协调模式。它也不是专门为 cross-embodiment transfer 设计的，因为视觉对齐依赖 unimanual/bimanual 观察之间的相似性。还有一个真实失败模式很值得注意：小的 end-effector deviation 会导致抓取失败；严重旋转预测会让 low-level planner 难以处理，可能造成物体掉落。这些失败正好说明了我们为什么需要 action-level recovery augmentation。Skill transfer 能给模型一个好的初始能力，但闭环鲁棒性仍然需要偏离状态和纠正动作。

如果我们借鉴 AnyBimanual，具体可以做三件事。

第一，建立双臂 skill taxonomy。不要只按任务名组织数据，而要按动作块组织：

```text
single_arm_reach
single_arm_grasp
single_arm_place
stabilize_object
coordinated_lift
coordinated_push
handover_send
handover_receive
align_insert
twist_hold
twist_rotate
pour_hold
pour_tilt
```

每条轨迹的每个 block 都标注左右臂角色：active、passive、stabilizer、receiver、symmetric、leader、follower。即使先用规则标注，也比完全没有强。

第二，在 action 扰动时按 skill 角色采样。比如 `stabilize_object` 块，稳定臂的扰动要小，操作臂可以稍大；`coordinated_lift` 块，两臂扰动要共享高度和方向；`handover` 块，发送臂和接收臂的相对 pose 是关键；`twist` 块，固定臂保持 bottle frame，旋转臂沿 screw axis 做相位扰动。这样增强样本不会破坏 task semantics。

第三，把 policy 输出做成 keypose + planner 模式。对很多双臂任务，尤其是当前仿真平台里长程任务，直接连续控制可能难调。可以让模型预测：

```text
left_next_keypose_in_object_frame
right_next_keypose_in_object_frame
left_gripper
right_gripper
phase_transition_probability
```

然后用 IK / cuRobo / motion planner 执行到 keypose。扰动时，也在 keypose 上做，验证后再生成训练样本。这样比低层 action noise 更容易保证物理可行。

AnyBimanual 对我们最大的启发是：**双臂泛化不只是连续控制问题，也是技能组合和左右臂角色调度问题**。如果 action 扰动没有角色意识，就会生成“看起来多样，但技能语义错乱”的数据。真正有用的扰动应该在 skill block 内保持角色关系，只扩大状态覆盖和纠错能力。

## 6. ACT / ALOHA：双臂闭环 BC 的基础，action chunking 为什么有效

ACT / ALOHA 是 2023 的工作，不是最近一年，但它是双臂 imitation learning 里必须理解的基础。很多后续双臂论文都直接用 ACT 当 baseline 或 backbone，包括 D-CODA。它的核心贡献之一是 Action Chunking with Transformers：不是每次只预测下一步动作，而是预测未来一段 action sequence。ALOHA 系统用低成本双臂硬件采集细粒度双臂任务，比如开透明调料杯、插电池等，报告用很少示教能达到较高成功率。对我们讨论 action 扰动，它提供了一个基础问题：为什么 action chunking 本身能提高闭环成功率？

普通 BC 单步预测有两个问题。第一，双臂细粒度任务里动作很连续，单步监督会导致高频抖动。第二，人类示教有多模态和时序不一致，同一个视觉状态附近可能有不同微小动作，单步 L2 容易平均成不好的动作。ACT 预测一段动作，相当于让模型在更长时间尺度上输出一个连贯意图。执行时通常用 temporal aggregation 或 receding horizon，把多个预测 chunk 平滑融合。这样轨迹更平滑，双臂同步更好。

但是 action chunking 也有风险。chunk 越长，越像开环。双臂任务中，一旦物体实际运动和预测不一致，长 chunk 可能继续执行错误动作，把偏差放大。GLIDE 的消融也支持这一点：chunk 太短不平滑，太长反馈不及时。ACT 的正确使用方式不是“预测长 chunk 然后闭眼执行到底”，而是“预测 chunk，但频繁重算，只执行前面一小段，并做 temporal smoothing”。这点对 action 扰动很关键：如果你的训练数据只包含干净 chunk，模型在 rollout 时一旦 chunk 前几步造成偏差，后面就危险；如果你加入 perturbed-state -> recovery-chunk 数据，ACT 类模型才会在重新观察后输出纠正 chunk。

ACT 的另一个启发是，双臂 action label 本身要时间对齐。两只手的动作如果在数据采集里不同步，模型会学到错误相位。比如一只手已经夹住物体，另一只手还在靠近；示教延迟不一致时，监督 label 可能让模型一会儿等左手，一会儿等右手，闭环就会卡顿。action perturbation 里也一样：你可以扰 gripper timing，但必须让 recovery label 处理这个 timing error。比如“右手早夹 2 step”后的 recovery 可能是重新张开并对齐，而不是继续执行原 chunk。

如果要基于 ACT 做双臂 action 扰动增强，我建议这样设计。

第一，所有训练样本都用 chunk label：

$$
obs_t -> [a_t, a_{t+1}, ..., a_{t+H-1}]
$$

其中 `a` 是左右臂动作拼接，最好是 TCP delta 或 residual joint，而不是 absolute joint。H 可以从 16、20、32 开始 sweep。不要一开始用过长 chunk。

第二，扰动也按 chunk 做，而不是单步做。比如在 `t` 时刻对左/右 TCP 注入一个小偏差，然后 rollout 3 到 5 步，得到 `obs'_t`，再从 oracle 生成恢复 chunk `a'_{t:t+H}`。这样模型学到的是“从偏离状态开始的一段协调修正动作”，而不是一个瞬时纠错。

第三，设计 temporal perturbation。双臂失败经常来自相位错，不只是空间错。可以制造：

```text
left arm delayed by 1-3 steps
right arm delayed by 1-3 steps
gripper close early / late
one arm reaches pre-grasp early and waits
object moves while one arm has not updated
```

这些扰动比纯 xyz 高斯更贴近真实闭环错误。relabel 时要生成合理的等待、重新对齐、重新夹取动作。

第四，训练时对 recovery chunk 做 temporal aggregation 兼容。增强样本如果动作突然很大，ACT 可能通过 temporal aggregation 平滑掉，导致纠正不够；如果动作过猛，又会真实执行危险。可以给 recovery chunk 加速度/jerk regularization，或者在 label 生成时限制每步最大 TCP delta、最大旋转、最大 gripper switching frequency。

第五，评估时不要只看 episode success。还要统计：

```text
mid-rollout perturb recovery rate
average recovery time
number of regrasp attempts
left-right synchronization error
object slip / drop rate
action jerk
planner failure rate
```

ACT 类模型常常 clean rollout 很好，但遇到中途偏差就不行；这些指标能直接看到 action augmentation 是否真的提高闭环鲁棒性。

从 embodied-arena 角度，ACT/ALOHA 最值得吸收的是模型接口和数据组织方式。我们可以先把每个双臂任务导出为标准 chunk dataset，再把 D-CODA / BiDemoSyn / GLIDE 风格的增强样本插进去。不要先纠结大模型。一个结构清楚的 ACT baseline，加上高质量 recovery chunks，往往比一个复杂但数据标签混乱的模型更容易跑出闭环提升。

ACT 给我们的最大启发是：**双臂动作泛化要在时间尺度上建模，不能只做单步 action noise**。闭环成功率来自两件事的平衡：chunk 提供平滑协调，receding horizon 提供反馈修正，perturbed recovery data 提供偏离后的动作标签。

## 综合建议：在 embodied-arena 里应该怎么做

如果目标是“通过 action/trajectory 扰动提高双臂闭环成功率”，我建议不要一开始做一个大而全的 randomizer，而是按下面这条路线落地。

### 阶段 1：统一数据和 action schema

先把所有双臂 demo 导出成一个统一格式。每帧至少包含：

```text
left_tcp_pose_world
right_tcp_pose_world
left_tcp_pose_object
right_tcp_pose_object
left_gripper_state
right_gripper_state
left_joint_pos
right_joint_pos
object_pose
task_goal
phase
contact_state
action_abs_joint
action_delta_joint
action_tcp_delta_world
action_tcp_delta_object
success / failure / recovery flag
```

姿态统一用 Isaac Lab 新版 quaternion `[x, y, z, w]`，或者直接用 rotation 6D 避免符号问题。先不要只保存一种 label。不同论文对 action 表示的结论并不完全一样，保留多种 label 才能做真实消融。

### 阶段 2：phase-aware 扰动

把轨迹切成阶段。可以先规则化：

```text
approach: 没接触，距离目标变小
pre_contact: gripper 接近 affordance，准备闭合或接触
contact_establish: 刚闭合/刚接触
contact_transport: 双臂共同移动物体
precision_align: 插入/对孔/对轴/对容器
release: 放开或结束接触
retract: 回撤
```

每个阶段定义扰动：

```text
approach:
  TCP xyz 1-3cm
  yaw 3-10deg
  low-frequency drift

pre_contact:
  TCP xyz 0.5-2cm
  rotation 2-5deg
  gripper timing +/- 1-2 steps

contact_establish:
  grasp point 0.3-1cm
  close timing +/- 1-3 steps
  one-arm delay 1-2 steps

contact_transport:
  shared dual-arm rigid perturbation
  preserve left-right relative pose
  object-frame perturbation 0.5-2cm

precision_align:
  target-prepose perturbation
  insertion axis perturbation 1-3deg
  retreat-and-realign recovery

release/retract:
  small timing perturbation
  avoid collision / object drag
```

### 阶段 3：contact-aware 双臂约束

写一个 dual-arm perturbation validator。至少检查：

```text
IK valid
joint limits valid
left-right collision free
arm-scene collision free
TCP not below table
object still reachable
contact-rich phase preserves relative pose bounds
gripper command consistent with contact state
per-step delta below safety threshold
```

contactless 可以左右臂独立扰；contact-rich 要从 shared perturbation 或 object frame 扰。比如双臂托举时，左右 TCP 的高度变化应该一致；双臂推箱子时，左右 TCP 施力方向应该一致；拧瓶盖时，固定臂扰动小，旋转臂沿轴扰动。

### 阶段 4：relabel，而不是 noisy label

每个扰动状态都要重新生成 label：

```text
obs_perturbed -> recovery_action_chunk
```

relabel 来源可以按难度从低到高：

1. 最近未来 waypoint 对齐：从扰动状态规划回原轨迹未来某个 waypoint。
2. IK + object-frame keypose：根据 object pose 重新计算下一关键 pose。
3. task scripted oracle：针对具体任务写恢复逻辑。
4. cuRobo / motion planner：生成无碰撞恢复路径。
5. contact-aware optimizer：对接触阶段重新优化双臂轨迹。
6. human intervention：真实或仿真中人工纠正，作为最高质量 recovery 数据。

一开始建议用 1、2、3 就够了。复杂 contact-rich 再上 4、5。

### 阶段 5：训练和消融

不要一次把所有增强开满。按顺序做 ablation：

```text
baseline ACT / diffusion policy
+ residual action
+ action chunk sweep
+ approach/pre_contact perturb recovery
+ contact-rich shared perturb recovery
+ temporal delay / gripper timing recovery
+ block retargeting / object-centric augmentation
```

每一步只加一种机制，看闭环成功率和 recovery 指标是否提升。

### 阶段 6：评估要专门测闭环恢复

建议评估集分成：

```text
clean success
initial object pose OOD
mid-rollout object displacement
left arm delay
right arm delay
gripper early/late
contact slip
target pose perturbation
new object geometry
new task composition
```

核心指标：

```text
success rate
recovery success rate
mean recovery steps
object drop/slip rate
left-right relative pose error
action jerk
IK/planner failure rate
phase completion rate
```

如果只看 clean success，很多 action 扰动的收益看不出来。真正的目标是：policy 犯了小错之后，能不能自己救回来。

## 最推荐的落地组合

如果只能选一个最小可行方案，我建议这样做：

1. 用 ACT 或现有 diffusion policy 做 baseline，action 改成 residual / object-centric TCP delta chunk。
2. 轨迹切 phase，先从 `approach`、`pre_contact`、`contact_transport` 三个阶段做扰动。
3. contactless 阶段独立扰左右 TCP；contact-rich 阶段扰 shared dual-arm frame，保持左右相对 pose。
4. 用 IK + task oracle 生成 recovery chunk，失败样本丢弃。
5. 原数据和 recovery 数据混合训练，recovery loss 权重 2 到 5 倍。
6. 单独做 mid-rollout perturb eval。

这条路线本质上是：

```text
D-CODA 的 contact-aware perturb + relabel
GLIDE 的 residual/chunk action 表示
BiDemoSyn 的 block-level 任务语义保持
ACT 的 action chunking 和闭环重预测
```

这比“随机给 action 加高斯噪声”要复杂一些，但这是双臂任务里真正可能提高闭环成功率的方向。
