# 前置知识：行为克隆 → RL 微调范式（完整理解）

> **为什么要读这篇**：DPPO 的核心是"先 BC 预训练，再 RL 微调"。RLHF 也是同一套思路。不懂这个范式，就不理解为什么需要 DPPO、为什么不直接从零用 RL。  
> **涉及概念**:  
> - 行为克隆 (Behavior Cloning, BC)  
> - 分布偏移 (Distribution Shift)  
> - 模仿学习 (Imitation Learning)  
> - RLHF (Reinforcement Learning from Human Feedback)  
> - 预训练 + 微调 (Pre-training + Fine-tuning)  
> - 策略坍塌 (Policy Collapse)

## 第一部分：行为克隆——最简单的策略学习

### 1.1 问题设定

你有一个"专家"（比如人类遥操作者），他在各种情况下做出了正确的动作。你把这些 (观测, 动作) 对收集起来，形成一个数据集。

```text
数据集 D = {(o₁, a₁), (o₂, a₂), ..., (o_N, a_N)}

其中:
  oᵢ: 第 i 个时刻的观测（图像+关节角度）
  aᵢ: 专家在那个时刻做的动作
```

### 1.2 训练过程

就是标准的监督学习：

$$
对连续动作空间:
  Loss = E_{(o,a)~D} [‖π_θ(o) - a‖²]    # MSE 回归
  
对 Diffusion Policy:
  Loss = E_{(o,a)~D, k, ε} [‖ε - ε_θ(a_k, k, o)‖²]  # 噪声预测
  
对离散动作空间 (LLM):
  Loss = E_{(o,a)~D} [-log π_θ(a|o)]    # 交叉熵

都是监督学习。输入观测，输出动作，用 ground truth 作为标签。
$$

### 1.3 BC 的优势（为什么它是起点）

```text
1. 简单: 标准监督学习，所有深度学习工具直接适用
2. 稳定: 不像 RL 那样有训练发散的风险
3. 数据高效: 50-300 条轨迹就能学到像样的策略
4. 离线: 不需要仿真器或真实环境在训练时运行
5. 快速: 几个小时训完（vs RL 可能要几天几周）
6. 可解释: 策略做了什么很容易理解（它在模仿示教）
```

## 第二部分：行为克隆的致命缺陷

### 2.1 分布偏移（Distribution Shift）——核心问题

```text
训练时: 策略只见过专家走过的状态分布 d_expert(s)
部署时: 策略自己执行，一旦犯错 → 进入从未见过的状态

数学表述:
  训练数据的状态分布: s ~ d_expert
  部署时的状态分布:   s ~ d_π (由策略自己的行为决定)
  
  如果 d_π ≠ d_expert → 策略的行为在新状态上没有保障
```

**直觉理解（开车类比）**：

```text
训练: 你看了 100 小时教练在正常道路上开车的视频
       → 你学会了 "在正常路况下怎么开"

部署: 你自己开车
  第 1 秒: 正常路况 → 你做得对（和教练一样）
  第 2 秒: 稍微偏了一点点（手有点抖）
  第 3 秒: 偏得更多了（因为第 2 秒没完全纠正）
  ...
  第 10 秒: 已经偏出正常行驶区域 → 你从没见过这种情况 → 完全不知道怎么做
  第 11 秒: 撞了
```

### 2.2 误差的复合增长

$$
假设每步策略有概率 ε 犯小错（偏离专家轨迹一点点）

T 步后的总偏差:
  最坏情况: ~ O(ε × T²) 的误差累积（二次增长！）
  
  为什么是二次而不是线性:
    t=1: 偏差 ε → 进入稍微不同的状态
    t=2: 在新状态上犯的错可能比 ε 更大（因为状态已经不在训练分布里了）
    → 误差累积效应
    
  100 步的任务: 总偏差 ~ 100² × ε = 10000ε
  如果 ε = 0.01: 总偏差 = 100 → 完全不可控
$$

### 2.3 BC 的天花板

即使完美解决了分布偏移（比如有无限数据覆盖所有状态），BC 还有一个根本限制：

**策略最好也只能和专家一样好。不能超越。**

$$
为什么不能超越:
  BC 的目标: min ‖π(o) - a_expert‖²
  最优解: π(o) = a_expert（完美模仿）
  
  但如果专家本身的动作不是最优的呢?
  - 人类手抖 → 策略也学会抖
  - 人类犹豫 → 策略也犹豫
  - 人类用了次优路径 → 策略也用次优路径
  
  BC 不知道什么是"好"什么是"坏" → 它只知道"模仿"
$$

## 第三部分：为什么不直接从零用 RL

### 3.1 RL 的理论优势

RL 通过和环境交互 + 奖励反馈来优化策略。理论上可以找到超越任何专家的最优策略。

$$
BC: "模仿别人" → 上限 = 示教质量
RL: "自己探索" → 上限 = 任务的物理极限
$$

### 3.2 但 RL 从零开始几乎不 work（在复杂机器人任务上）

```text
原因 1: 探索困难
  14 维连续动作空间 + 50Hz 控制频率
  随机探索碰到有意义的行为的概率 ≈ 0
  
  类比: 让一个从没摸过琴的人随机按钢琴键，
        期望他碰巧弹出一首完整的曲子 → 不可能
  
原因 2: 稀疏奖励
  很多任务只有完成时才给奖励:
    "把螺母拧上螺栓" → 只有完全拧上才得 1 分
    随机探索永远碰不到这个奖励 → RL 没有学习信号
    
原因 3: 时间开销
  简单的 Gym 任务（Hopper, Walker）: 从零 RL 需要 100 万步
  复杂操作任务: 需要 1000 万+ 步
  每步需要仿真或真机交互 → 时间成本巨大
  
原因 4: 安全风险
  真实机器人上从零探索 → 会做出各种危险动作
  撞坏设备、伤到人 → 不可接受
```

### 3.3 定量对比

```text
FurnitureBench One-leg 任务:
  - BC (Diffusion Policy): 50 条示教，训练 8 小时 → 成功率 65%
  - 从零 RL (PPO): 10000 万步，训练 100+ 小时 → 成功率 0%（太难探索）
  - BC + RL微调 (DPPO): 50 条示教 + 10 小时微调 → 成功率 95%

Robomimic Transport (双臂操作):
  - BC: 300 条示教 → 成功率 ~60%
  - 从零 RL: 无论多少步 → 成功率 0%（14维稀疏奖励不可能从零学）
  - BC + RL微调: 300 条示教 + 微调 → 成功率 >90%
```

## 第四部分：预训练 + 微调 = 最优范式

### 4.1 核心思路

```text
阶段 1 (BC 预训练):
  用专家数据训一个"还行"的初始策略
  → 策略能完成任务 50-70% 的情况
  → 建立了合理的动作分布（不会做完全离谱的事）

阶段 2 (RL 微调):
  从 BC 策略出发，用 RL 继续优化
  → 策略不需要从零探索（BC 已经给了好起点）
  → 只需要在"差不多对"的附近做精细调整
  → 通过奖励反馈学到超越示教的行为
```

### 4.2 为什么两阶段比单独任何一个强

```text
只用 BC:
  ✓ 快速得到一个可用策略
  ✗ 受限于示教质量
  ✗ 分布偏移问题
  ✗ 不能超越专家

只用 RL:
  ✓ 理论上能找到最优策略
  ✗ 从零探索效率极低
  ✗ 稀疏奖励几乎不可能学到
  ✗ 安全风险大

BC + RL:
  ✓ BC 提供好起点 → 避免从零探索
  ✓ RL 提供超越能力 → 打破 BC 天花板
  ✓ RL 只需局部探索 → 高效安全
  ✓ 两者优势互补
```

### 4.3 类比：学打篮球

```text
方案 A: 只看视频学（BC）
  看了 1000 小时 NBA 比赛视频
  → 知道大概怎么运球、投篮
  → 但自己上场时动作走形、没有实战感觉
  → 天花板: 知道"应该怎么做"但做不到完美

方案 B: 不看视频直接上场乱打（从零 RL）
  自己瞎投 10 万次
  → 前 9 万次几乎全不进（没有方向）
  → 极慢地摸索出一些模式
  → 可能学出奇怪的姿势（local optima）

方案 C: 先看视频再上场练习（BC + RL）
  先看视频学到基本姿势（BC）
  然后上场实践 + 根据进球反馈调整（RL）
  → 很快就能在正确基础上精进
  → 最终超越视频中的水平
  → 这就是人类实际的学习方式
```

## 第五部分：RLHF——同一范式在 LLM 中的应用

### 5.1 LLM 的预训练 + 微调 Pipeline

```text
阶段 1: 预训练 (无监督)
  在万亿 token 上做 next-token prediction
  → 学会语言的统计规律
  → 但不一定"有用/安全/对齐人类意图"

阶段 2: SFT (Supervised Fine-Tuning)
  在高质量 (指令, 回答) 对上做监督微调
  → 学会对话格式和指令遵循
  → 这一步相当于机器人中的 BC

阶段 3: RLHF
  a. 训练奖励模型 (Reward Model):
     给同一个 prompt 的多个回答排序 → 学一个打分器
     
  b. 用 PPO 微调 LLM:
     prompt → LLM 生成回答 → 奖励模型打分 → PPO 更新 LLM
     
  → 让回答更 helpful、harmless、honest
  → 这一步相当于机器人中的 DPPO/RL 微调
```

### 5.2 机器人 vs LLM 的对应关系

```text
                    LLM 领域                机器人领域
预训练基座:         GPT (next-token)        不适用
模仿学习:          SFT (指令微调)          BC (Diffusion Policy)
奖励来源:          人类偏好排序 → RM       环境奖励 (task success)
RL 算法:           PPO                     PPO / DPPO
防止偏离约束:      KL penalty to SFT       PPO clip / 结构化探索
最终目标:          超越 SFT 的"有用性"    超越 BC 的"成功率"
```

### 5.3 共同的核心挑战：策略坍塌

```text
问题: RL 微调容易让策略"跑偏"

LLM 中的策略坍塌:
  - 模型发现重复某些词能获得高奖励 → 疯狂重复
  - 模型发现极端长/短回答能 hack 奖励模型 → 长度失控
  - 回答变得奇怪但奖励高 → "reward hacking"

机器人中的策略坍塌:
  - 策略发现某些动作序列能偶然触发奖励 → 动作变得诡异
  - Gaussian 策略 PPO 微调后动作高频抖动 → 仿真里 work 但真机崩
  - 多模态策略坍缩成单模态 → 失去备选方案

共同解决方案: 限制微调后的策略不能偏离预训练太远
  LLM:  KL(π_RL ‖ π_SFT) penalty → 加在 reward 里
  DPPO: PPO clip + 扩散的结构化探索 → 天然在数据流形上
```

## 第六部分：RL 微调的具体挑战

### 6.1 奖励设计

$$
RL 需要奖励信号。但机器人任务的奖励很难设计:

稀疏奖励:
  "完成了=1, 没完成=0"
  优点: 不需要手工设计中间步骤
  缺点: 信号太弱，策略不知道"朝哪个方向改进"
  
  例: 组装家具的奖励
    稀疏: 整个组装完成才得分 → 从零几乎学不到
    DPPO 能用: 因为 BC 预训练已经接近成功，微调只需要精细调整

密集奖励:
  每步给中间信号（如 -距离目标 + 每个子步骤 bonus）
  优点: 学习信号强，收敛快
  缺点: 需要人工设计每个子步骤 → 工程量大 → 容易 reward hack
  
  例: 抓取物体的密集奖励
    r = -距离 + 到达物体上方×0.5 + 接触×1.0 + 抬起×2.0 + 放到目标×5.0
    设计这些中间奖励需要对任务有深入理解

DPPO 的选择: 几乎全部使用稀疏奖励
  原因: BC 预训练让策略已经"接近成功"
        微调时只需要稀疏信号就够了
        避免了密集奖励设计的工程成本
$$

### 6.2 On-Policy vs Off-Policy

```text
On-Policy (PPO, DPPO):
  - 每次用当前策略收集数据
  - 数据用完就扔（不存历史数据）
  - 新数据来自当前策略 → 梯度方向正确（无分布偏移）
  
  优点: 训练稳定
  缺点: 样本效率低（数据只用一次）
  适合: 有大量并行环境（GPU 仿真，几千个环境同时跑）

Off-Policy (DQL, IDQL, SAC):
  - 数据存在 replay buffer 里反复使用
  - 可以用过去任何策略收集的数据
  
  优点: 样本效率高
  缺点: 训练不稳定（旧数据的分布和当前策略不匹配）
  适合: 数据稀缺场景（真实机器人只有 1 个环境）

DPPO 选了 on-policy + 大规模并行:
  1000-4096 个 GPU 并行环境 × PPO
  → 虽然每批数据只用一次，但每次有几千条轨迹
  → 实际上不缺数据
```

### 6.3 对扩散策略的特殊难题

$$
难题 1: log π(a|s) 不可算
  高斯策略: log N(a; μ, σ²) → 解析公式
  扩散策略: log p(a₀|s) = log ∫...da₁...da_K → 不可解析
  
  DPPO 的解法: 展开去噪链，对每步用高斯 log-likelihood

难题 2: 多步去噪 → 等效 horizon 变长
  高斯策略: T 步 episode → PPO 处理 T 步
  扩散策略: T 步 episode × K 步去噪 → PPO 处理 T×K 步
  
  horizon 变长 → credit assignment 更难 → 梯度方差增大
  
  DPPO 的解法: γ_denoise < 1 折扣早期步 + 只微调后几步

难题 3: 多模态保持
  RL 微调自然倾向让策略变得更"尖锐"（更确定性）
  多模态策略 → RL 微调 → 可能坍缩成单模态
  
  DPPO 的自然优势: 扩散过程从噪声开始 → 天然保持采样多样性
  但还是需要注意: 过度微调会让多模态性下降
$$

## 第七部分：微调时防止策略偏离的方法

### 7.1 为什么需要约束

```text
RL 只追求奖励最大化。如果不加约束:
  - 策略可能学到 reward hacking（不完成任务但触发奖励）
  - 策略可能变得极端（如高频抖动在仿真中有效但真机不行）
  - 策略可能丢失预训练学到的有用行为

需要在"改进"和"不偏离太远"之间平衡。
```

### 7.2 各种约束方法

**方法 1：KL Penalty（RLHF 中的标准做法）**

```text
reward_modified = reward - β × KL(π_current ‖ π_pretrained)

β 控制约束强度:
  β 大: 策略很难偏离预训练 → 改进有限
  β 小: 策略容易偏离 → 可能坍塌

LLM RLHF 中常用。对每个 token 计算 KL。
```

**方法 2：PPO Clip（DPPO 的选择）**

$$
clip(r(θ), 1-ε, 1+ε) × Advantage

每次更新策略变化不超过 ε:
  ε=0.2: 概率最多变化 20%
  ε=0.01: 概率最多变化 1%（非常保守，DPPO 操作任务用这个）
  
优点: 不需要计算 KL（计算更快）
     隐式限制了总偏离量（多次小步 ≠ 一次大步）
$$

**方法 3：BC 正则损失**

$$
Loss = RL_loss + α × BC_loss
     = PPO_objective + α × E[-log π_pretrained(a|s)]

让新策略在预训练策略下也有高似然。
DPPO 论文中 Gaussian 基线加了 BC 正则:
  仿真 53% + 真机 50%（有效但限制了改进幅度）
$$

**方法 4：扩散模型的天然约束（DPPO 独有优势）**

```text
扩散模型的去噪过程天然把采样拉向数据流形:
  即使 PPO 更新了参数 → 去噪过程仍然会把动作"修正"回合理范围
  
  类比: 高斯策略加噪声 → 噪声直接作用在最终动作上
       扩散策略加噪声 → 噪声经过多步去噪后被"洗掉"大部分

  这就是为什么 DPPO 不需要额外的 KL penalty 或 BC 正则:
  扩散模型本身就是约束。
```

## 第九部分：思考题

1. DAgger（让专家标注策略遇到的新状态）和 RL 微调都是解决分布偏移。它们的本质区别是什么？
   → DAgger: 需要专家持续在线 → 策略最好等于专家（不能超越）
   → RL 微调: 不需要专家，用环境奖励 → 可以超越专家

2. 为什么 RLHF 要加 KL penalty 而 DPPO 不需要？
   → LLM 的 token 空间是离散的、巨大的，策略可以轻易跑到奇怪的区域
   → 扩散策略的去噪过程天然把采样约束在数据流形上
   → DPPO 的 clip ε=0.01 已经非常保守（比 LLM RLHF 保守得多）

3. Off-policy 方法（DQL）样本效率更高，为什么 DPPO 论文说 on-policy 更好？
   → 因为有 1000+ 并行环境时，on-policy 数据不缺
   → off-policy 的不稳定性（Q 函数估计偏差）在稀疏奖励+高维动作下是致命的
   → 最终效果: 稳定 > 样本效率

4. 如果有无限的专家数据（覆盖所有可能状态），还需要 RL 微调吗？
   → 覆盖所有状态 → 分布偏移问题消失 → BC 可以 work
   → 但仍然不能超越专家（如果专家不是最优的）
   → 想要超越 → 仍然需要 RL
   → 但实际中"无限数据覆盖所有状态"是不可能的

5. 为什么 DPPO 微调后的策略在 Sim-to-Real 中表现反而比 Gaussian PPO 好？
   → Gaussian PPO 微调后动作高频抖动（在仿真完美执行器下 work）
   → 真机执行器有带宽限制 → 高频被滤掉 → 实际轨迹偏离 → 失败
   → DPPO 的多步去噪天然是 low-pass filter → 输出平滑 → 真机 work
