# 写作规范

## 0. 项目愿景

**本项目的终极目标**：让一个零基础的小白，完全在这一个网站上就能学会 RL、深度学习、以及机器人学习的所有相关知识。读者不需要翻论文原文、不需要找其他教程——这里就是他的唯一学习入口。

为了实现这个愿景，必须遵守以下核心原则：

### 知识自包含原则

- **遇到任何超出大学本科水平的概念，必须自动写一篇前置知识去解释它**。不能假设读者知道 KL 散度、ELBO、GAE、Actor-Critic 是什么——每一个都要有独立的前置知识文章。
- **判断标准**：如果一个概念对"有基本线代和概率知识的本科生"来说需要学习，就必须有前置知识文章。
- **执行方式**：写任何文章时，遇到需要解释的概念，如果已有前置知识就链接过去；如果没有，**立刻创建**对应的前置知识文件（哪怕先写个框架后面再填充），绝不留空白。

### 前置知识的原子性与无重复原则

- **原子化**：一篇前置知识只讲**一个**概念。"SAC"是一篇、"DDPG"是一篇、"Q 函数"是一篇、"Replay Buffer"是一篇。绝不在一篇里塞两个独立概念。
- **无重复**：同一个概念在整个项目中只有**一个**权威讲解位置。如果"Q 函数"在 SAC 文章中需要解释，不要在 SAC 文章里重新讲一遍 Q 函数——链接到 Q 函数的前置知识文章就行。
- **粒度判断**：如果一个子概念在多篇文章中被反复解释，说明它应该被提取为独立的前置知识。反之，如果一个概念只在一篇文章中出现一次且 3 句话能说清，可以内联解释不用单独成文。

### 知识图谱式的链接网络

项目的所有文章构成一个**有向无环图（DAG）**：
- 前置知识 → 前置知识（如"DDPM" → "对数似然与 ELBO"）
- 前置知识 → 论文精读（如"PPO" → "DPPO 精读"）
- 论文精读 → 综述（如"DPPO 精读" → "扩散策略 RL 综述"）

读者可以从任何一个节点进入，顺着链接探索整个知识体系，永远不会遇到"这个概念没解释"的断点。

### ⚠️ 知识全阶段覆盖原则（极其重要）

**任何一个知识点，必须覆盖从原理 → 概念 → 具体应用的完整链路，一个阶段都不能少。**

具体来说，当介绍一个新概念/方法时，必须包含以下四个阶段（可以在同一篇文章中，也可以拆分成多篇互相链接）：

1. **基础原理阶段**：这个概念依赖的底层数学/物理直觉是什么？
   - 例：介绍 LoRA 时，不能跳过"矩阵的秩"这个概念。必须有一篇独立的前置知识解释秩是什么、为什么低秩矩阵可以压缩。
   
2. **概念/算法阶段**：这个方法本身的设计是什么？公式是什么？为什么这样设计？
   - 例：LoRA 的 $\Delta W = BA$，$\alpha/r$ 缩放，初始化策略等。

3. **工程实操阶段**：具体怎么用？模型怎么改？代码怎么写？训练流程是什么？
   - 例：LoRA 怎么注入模型、训练时梯度怎么流、部署时怎么合并。必须有可运行的代码示例。
   - **这一阶段绝不能省略！** 很多读者在"公式我看懂了但不知道怎么实际操作"这一步卡住。

4. **应用场景阶段**：在真实项目中怎么用？有什么最佳实践？踩过什么坑？
   - 例：LoRA 在 VLA 中怎么配置、$r$ 怎么选、哪些层加哪些层不加、和 RL 怎么配合。

**执行方式**：
- 写任何概念时，自查这四个阶段是否都有对应的文章覆盖
- 如果某个阶段缺失，立即创建对应文章（哪怕先写个骨架）
- 在文章中用链接把四个阶段串联起来，让读者能顺畅地从原理一路读到应用

**反面例子（禁止）**：
- 只讲公式不讲怎么用 → 读者知道 $\Delta W = BA$ 但不知道代码怎么写
- 只讲代码不讲原理 → 读者会用但不知道为什么这样用
- 只讲概念不讲应用场景 → 读者不知道在自己的项目中该怎么配置

**正面例子（要求）**：
LoRA 知识体系应该有这样的完整链路：
- [矩阵的秩与低秩近似](/前置知识/000z_前置知识_矩阵的秩与低秩近似)（原理阶段）
- → [LoRA 低秩适配基础](/前置知识/000x_前置知识_LoRA低秩适配基础)（概念阶段）
- → [LoRA 实操全流程](/前置知识/001a_前置知识_LoRA实操全流程_模型如何支持LoRA)（工程实操阶段）
- → [LoRA 在 VLA 中的应用](/论文综述/067_LoRA_VLA_机器人视觉语言动作模型适配)（应用场景阶段）

读者沿着这条链路读下来，从"秩是什么"一直读到"在我的机器人项目里 r 应该设多少"，没有任何断点。

## 1. 面向读者：小白友好 + 有深度

### 核心原则

- **假设读者是聪明的本科生**：有基本数学素养（线代、概率），但不了解具体领域。任何超出这个基线的知识（如 KL 散度、策略梯度、扩散模型、变分推断），都必须有前置知识文章兜底
- **每个公式必须"拆碎了讲"**：不能只列公式，必须逐项解释每个符号的含义、为什么是这个形式。**公式出现之前必须先用自然语言交代"这个公式是用来干什么的"**——读者必须在看到公式之前就知道它要解决什么问题。绝不允许毫无铺垫地甩出一个公式。
- **先直觉再形式**：任何概念先用一句话说明"在干什么"，再给严格定义
- **举例子！举例子！举例子！**：最好有一个**贯穿全文的具体例子**（如"一个 2D 桌面上的机器人抓取"），每次引入新概念时都用这个例子来解释
- **⚠️ 必须比原论文更详细**：不论是精读还是综述，写出来的内容必须比原论文更加详细、更加易懂。原文一笔带过的地方，我们要展开讲清楚；原文没有例子的地方，我们要补充具体数值例子；原文跳步的推导，我们要补全中间步骤。**绝不允许一句话概括一个方法就跳过**——如果一个概念值得提，就值得花几段话把它讲透。这是本项目存在的意义：让读者读我们的文章就够了，不需要再回去看原文。
- **⚠️ 新概念/组件引入必须先建立连接**：任何新出现的概念、模块或组件，必须在引入时**立刻回答三个问题**：
  1. **它和已有系统是什么关系？**——比如"这个模块插在 VLA 的 Transformer backbone 之后、action head 之前"，或者"它替换了原来的 xxx 模块"。必须让读者能在脑中把新概念"放进"已有的系统架构图里。
  2. **在什么阶段使用？**——是训练时用？推理时用？两者都用？在整个 pipeline 的第几步？
  3. **为什么需要它？设计初衷是什么？**——解决了什么之前解决不了的问题。
  
  **反面例子**（禁止）：直接说"TOPIC 由三个组件构成：Task-Specific Prompt Pool、Task Relation Graph、Graph-guided Initialization"——读者读到这里完全不知道这些东西和 VLA 模型是什么关系，在哪里插入，什么时候用。
  
  **正面例子**（要求）：先说"标准 VLA 在顺序学习新任务时，新任务的梯度会覆盖旧任务的参数→遗忘。一种解法是：冻结 VLA 的整个 backbone，改为在输入序列前面插入一小组可学习向量（称为 prompt），每个任务只训练各自的 prompt（约 80K 参数）。这样不同任务的参数完全隔离，就不存在覆盖问题。"——读者立刻明白了 prompt 在系统中的位置（输入端）、阶段（训练时学习，推理时选择）、和 VLA 的关系（backbone 冻结，prompt 是唯一可变部分）。
- **⚠️ 遇到未覆盖的概念必须主动补充**：写文章过程中，一旦发现引用了一个还没有对应前置知识的概念（如提到 DDPG 但没有 DDPG 前置文章），必须**立即创建**那篇前置知识（至少创建文件+写好骨架），不能只留 TODO 不管。项目的完整性比任何单篇文章的完成度更重要。
- **⚠️ 代码必须"先讲后贴"，禁止直接甩代码**：任何代码块出现之前，必须先用自然语言讲清楚这段代码**要做什么、为什么这样做、核心思路是什么**。读者应该在看到代码之前就已经理解了设计意图，代码只是"验证"他的理解。具体要求：
  1. **代码前必须有文字铺垫**：至少 2-3 句话说明"接下来这段代码要实现什么功能"、"核心思路是 xxx"。
  2. **禁止连续贴多段代码**：两段代码之间必须有文字解说，解释前一段代码的关键点、引出后一段代码的动机。
  3. **代码中的关键行必须有注释或紧跟讲解**：如果一行代码的含义不显而易见（如 `torch.bmm(x, selected_W)`），必须在代码注释中或代码块后面解释它在做什么。
  4. **先概念后实现**：引入一个新模块时的正确顺序是——(a) 这个模块解决什么问题 (b) 它的设计思路是什么 (c) 具体实现（代码）(d) 代码中值得注意的细节。绝不允许 (a) 之后直接跳到 (c)。
  
  **反面例子**（禁止）：
  > "N1.7 的 MultiEmbodimentActionEncoder：" 然后直接贴 20 行代码。
  
  **正面例子**（要求）：
  > "N1.5 的问题是所有机器人共享一组权重——Franka 的'关节角'和 G1 的'左臂角度'虽然语义完全不同，却被迫通过同一个线性层映射。N1.7 的解决方案是：给每种机器人分配一组**独立的**权重矩阵。具体做法是把普通的 `nn.Linear` 替换为 `CategorySpecificLinear`——它内部存储了 N 组权重，前向传播时根据 `embodiment_id` 选择对应的那一组。来看实现："

### ⚠️ 公式讲解模板（最高优先级规则）

**每一个独立公式（$$...$$）后面都必须紧跟详细讲解，没有例外！** 尤其是长公式、多项式、含积分/期望/梯度的公式——越复杂的公式越需要更多篇幅的讲解。绝不允许写完公式就跳到下一个话题。

**这条规则适用于所有文章类型：前置知识、论文精读、综述。没有例外。**

每个重要公式后必须跟随以下**全部 5 项**（缺一不可）：

1. **为什么需要这个公式**（1-3 句话）：先交代动机——我们遇到了什么问题，为什么需要用数学来解决它。读者必须在看公式之前就知道"这个公式要帮我算什么"。
2. **一句话直觉**（必须用 > 引用格式）：用最通俗的语言说明这个公式在干什么，不用任何数学符号。比如"把新旧策略的差异限制在一个安全范围内"。
3. **逐项拆解**（用表格或列表）：公式中**每一个符号**都必须解释。不能假设读者知道 $\nabla_{\mathbf{x}}$、$\mathbb{E}$、$\mathrm{div}$ 是什么。每个符号要说明：(a) 数学含义 (b) 物理/几何直觉 (c) 在这个具体场景中对应什么。
4. **具体数值例子**（必须有具体数字）：假设维度 $d=2$ 或 $d=3$，代入实际数字完整算一遍。不能说"代入数字即可验证"——必须真的写出来。数值例子是读者建立直觉最重要的工具。
5. **为什么是这个形式**（1-3 句话）：设计动机。为什么用 $\frac{1}{2}$ 而不是 $1$？为什么取 min 而不是 max？为什么用 MSE 而不是交叉熵？

**特别强调**：

- **长公式必须拆成多行讲解**：如果一个公式超过 5 个符号/项，必须把每一项单独拎出来解释，不能一股脑全堆在一个表格里。
- **含积分的公式必须解释积分的含义**：$\int_0^1 (\cdots)\mathrm{d}t$ 的物理含义是什么？积分变量是什么？上下限代表什么？
- **含期望 $\mathbb{E}$ 的公式必须说明"对谁求期望"**：期望的下标是什么分布？在实际训练中怎么近似这个期望（mini-batch？Monte Carlo？）？
- **含梯度 $\nabla$ 的公式必须解释"对谁求梯度"以及"梯度方向的物理含义"**
- **禁止"显然"、"容易验证"、"不难看出"等表述**：对读者来说没有任何东西是"显然"的。

### 前置知识引用原则

- **公式中出现的通用概念**（如 $Q(s,a)$、$V(s)$、advantage、GAE、KL 散度、交叉熵等），如果不是本文主题，**必须链接到前置知识文章**
- **前置知识应该原子化**：一篇前置知识只讲一个概念（如"Q 函数与 Value 函数"是一篇、"GAE 优势估计"是一篇），但必须讲透、讲明白
- **判断标准**：如果一个符号/概念需要超过 3 句话才能解释清楚，它就应该有自己的前置知识文章
- **行文中遇到需要前置知识但还没写的文章时**：先写一个占位链接标注 `<!-- TODO: 需要前置知识文章 -->`，然后尽快补充

### ⚠️ 前置知识的独立性原则（极其重要）

**前置知识是上游，论文精读/综述是下游。** 依赖关系只能单向流动：论文可以引用前置知识，但前置知识绝不能依赖某篇论文的上下文。

具体要求：
- **前置知识必须完全独立于任何论文**：讲 SAC 就讲 SAC 本身，不要说"为了训练 Residual 网络所以用 SAC"。这种应用场景的描述属于论文精读的内容。
- **前置知识的例子必须是通用的**：用"一个机械臂学习抓取"或"一个小车学平衡"这种通用场景，不要用某篇论文特有的场景（如"VLA 的 Residual 修正网络"）
- **前置知识的动机必须是概念本身的动机**：讲 SAC 的动机是"连续动作空间需要 sample-efficient 的 off-policy 方法"，而不是"Residual RL 需要用 SAC"
- **论文精读中再做关联**：在论文精读中写"本文使用 [SAC](/前置知识/000k_前置知识_SAC_Soft_Actor_Critic) 训练 Residual 网络，具体原因是……"

**判断标准**：如果删掉项目中所有论文精读和综述文章，前置知识文章读起来仍然完整、自洽、有价值——这就对了。前置知识是**教科书式的基础概念讲解**，不是某篇论文的配套说明。

示例：

```markdown
$$
L^{\text{CLIP}}(\theta) = \mathbb{E}\left[\min\left(r_t(\theta)\hat{A}_t, \; \text{clip}(r_t(\theta), 1-\epsilon, 1+\epsilon)\hat{A}_t\right)\right]
$$

**一句话**：这个损失函数让策略朝着"好动作"更新，但不允许一次更新太大。

**逐项拆解**：
- $r_t(\theta) = \frac{\pi_\theta(a_t|s_t)}{\pi_{\theta_{\text{old}}}(a_t|s_t)}$ — 新旧策略的概率比值，=1 表示没变
- $\hat{A}_t$ — 优势估计，正值说明这个动作比平均好
- $\text{clip}(r, 1-\epsilon, 1+\epsilon)$ — 把比值限制在 $[0.8, 1.2]$（$\epsilon=0.2$）
- $\min(\cdot, \cdot)$ — 取两者较小值，防止过于乐观的更新

**代入数字**：假设 $\hat{A}_t = +2$（这个动作很好），$r_t = 1.5$（新策略把这个动作的概率提高了 50%）：
- 未裁剪项：$1.5 \times 2 = 3$
- 裁剪项：$\text{clip}(1.5, 0.8, 1.2) \times 2 = 1.2 \times 2 = 2.4$
- 取 $\min(3, 2.4) = 2.4$ → 限制了过大更新

**为什么这样设计**：如果不裁剪，策略可能一步跳太远，进入不可恢复的差状态。
```

## 2. 交叉引用：文章之间必须互相链接

### 规则

- **前置知识引用**：文章中首次出现一个非本文主题的重要概念（如"PPO"、"DDPM"）时，必须用超链接指向对应的前置知识文章
- **前置知识之间也要互相引用**：前置知识不是孤岛！如果 SAC 文章提到了 DDPG / TD3，就必须链接到对应的前置知识。如果对应的前置知识还没写，先留 `<!-- TODO: 需要前置知识文章：DDPG/TD3 -->` 占位，但**绝不能只一句话带过不加链接**。前置知识之间的网状引用关系是帮助读者构建知识体系的关键——读者应该能从任何一篇入口，顺着链接阅读完所有相关前置知识。
- **论文之间引用**：综述引用精读、精读引用综述、综述之间引用，只要内容相关就加链接
- **知识链接区**：每篇文章开头必须有"相关阅读"区域，列出读本文前建议了解的内容

### 链接格式

跨目录用绝对路径：
```markdown
如果对 PPO 不熟悉，请先阅读 [策略梯度与 PPO](/前置知识/000a_前置知识_策略梯度与PPO)。
```

同目录用相对路径：
```markdown
更多细节参见 [DPPO 精读](./001_DPPO_扩散策略策略优化)。
```

### 行文中的自然引用

不要只在开头放链接，**正文中提到相关概念时也要加链接**：

```markdown
DPPO 的核心思路是对扩散过程的每一步应用 [PPO 的 clip 机制](/前置知识/000a_前置知识_策略梯度与PPO#53-ppoproximal-policy-optimization-2017)……
```

## 3. 文章结构

每篇文章必须包含：

1. **Frontmatter**（title、order、tags、category）
2. **标题 + 元信息**（论文标题、一句话概括）
3. **相关阅读区**：前置知识 + 关联文章的链接列表
4. **贯穿全文的例子定义**：在引言部分定义一个具体场景，后面反复用
5. **正文**：先直觉 → 形式化 → 例子 → 代码/图示
6. **总结与对比表**
7. **延伸阅读**：相关文章的链接

## 7. 篇幅要求

| 类型 | 最低字数 | 说明 |
|------|----------|------|
| 前置知识 | 3000+ 字 | 一个概念讲透 |
| 论文精读 | 5000+ 字 | 逐段解读 |
| 综述 | 8000+ 字 | 系统覆盖，大量例子和对比 |

综述不是"列举方法名"，而是让读者读完后**真正理解**每个方法在做什么、为什么这么做、和其他方法有什么区别。

- 综述：`S0N_综述主题名.md`
- 论文精读：`NNN_简称_中文主题.md`
- 系列文章：`系列/系列id/01_章节名.md`（详见第 10 节）

## 9. 发布

文章写好后放到对应目录即可，**侧边栏和标签页会自动生成**，无需手动编辑 config。唯一要确保的是 frontmatter 正确。

### 侧边栏行为

- **前置知识**：左侧显示完整的前置知识目录列表（按 category 分组）
- **论文精读 / 工程实践 / 工程项目**：左侧**不再**显示所有文章堆叠。每篇文章的左侧 sidebar 只显示：
  1. 当前文章标题
  2. 该文章的「知识链接」（从正文中 `**知识链接**：` 区域自动解析）
  3. 返回目录的快捷入口
- **系列文章**：左侧显示系列的完整章节目录

这意味着写文章时，`**知识链接**：` 区域非常重要——它既是读者的阅读地图，也是左侧导航的数据来源。

## 10. 系列文章

系列文章用于**长篇、多章节的系统性教程或深度专题**。与普通文章的区别：
- 首页/列表页只显示**一张卡片**（不是每章一张）
- 卡片上显示章节进度（如 `📖 2/5 章`）
- 左侧 sidebar 显示系列的完整章节目录
- 点击卡片进入系列概览页，再从概览页进入各章

### 10.1 目录结构

```
系列/
└── my_series_id/          # 系列 id（英文下划线命名）
    ├── index.md           # 系列概览/入口页（被卡片索引）
    ├── 01_第一章标题.md
    ├── 02_第二章标题.md
    ├── 03_第三章标题.md
    └── ...
```

### 10.2 系列概览页 `index.md` 的 Frontmatter

```yaml
---
title: "系列标题：完整教程名"
order: 500                    # 在卡片列表中的排序
tags: [标签1, 标签2, 系列]
category: 系列               # category 设为 "系列"
star: 5
series:
  id: my_series_id           # 与目录名一致
  totalChapters: 5           # 总章节数（必须手动维护）
  dir: /系列/my_series_id    # 章节所在目录路径
---
```

### 10.3 系列概览页 `index.md` 的正文内容

概览页应包含：
1. 系列简介（为什么写这个系列、适合什么读者）
2. **完整章节目录**（带链接和一句话简介）
3. 前置知识要求
4. 学习建议/阅读顺序

示例：

```markdown
# 从零实现 ACT：完整工程系列

> 从环境搭建到真机部署，手把手带你实现一个完整的 ACT 机器人控制系统。

## 章节目录

| 章节 | 标题 | 简介 |
|------|------|------|
| 01 | [环境搭建](./01_环境搭建) | Isaac Lab 安装、数据采集硬件配置 |
| 02 | [数据收集](./02_数据收集) | 遥操作方案选型、数据格式设计 |
| 03 | [模型搭建](./03_模型搭建) | ACT 网络结构实现、CVAE Encoder |
| 04 | [训练调优](./04_训练调优) | 超参搜索、训练技巧、过拟合处理 |
| 05 | [部署验证](./05_部署验证) | 推理优化、真机调试、成功率统计 |

## 前置知识

阅读本系列前建议了解：
- [行为克隆与 RL 微调范式](/前置知识/000d_前置知识_行为克隆与RL微调范式)
- [ACT Decoder 架构详解](/工程实践/ACT_Decoder架构详解)
```

### 10.4 章节文件的 Frontmatter

```yaml
---
title: "环境搭建"
series:
  id: my_series_id
  chapter: 1              # 章节编号，用于排序
order: 1                  # 也用于 sidebar 排序
---
```

章节文件**不需要**设置 `hidden: true`——它们在 `系列/` 子目录中，不会被 articles.data.mts 误扫描为独立卡片。

### 10.5 写系列文章的规范

系列文章的内容规范与普通文章完全一致（公式讲解、举例子、知识链接等），额外要求：

1. **每章开头有前情提要**：用 1-3 句话回顾上一章的结论，帮助读者衔接
2. **每章结尾有下章预告**：告诉读者下一步要解决什么问题
3. **每章独立可读**：虽然是系列，但每章应尽量自包含。读者跳读某一章时，不应因缺少上下文而完全看不懂
4. **交叉引用**：章节之间用相对链接互相引用（`./01_环境搭建`），引用外部知识用绝对链接
5. **totalChapters 必须维护**：新增或删除章节时，必须同步更新 `index.md` 中的 `series.totalChapters`

### 10.6 何时用系列 vs 普通文章

| 场景 | 选择 |
|------|------|
| 一个论文精读，5000 字左右 | 普通文章 |
| 一个概念讲解，3000 字 | 前置知识 |
| 一个完整工程项目教程，总计 2 万字以上 | **系列文章** |
| 一组相关论文的对比综述 | 普通综述 |
| 从零搭建某系统的完整过程，需要分步进行 | **系列文章** |
| 某技术的完整学习路径（如"从 Attention 到 VLA"） | **系列文章** |
